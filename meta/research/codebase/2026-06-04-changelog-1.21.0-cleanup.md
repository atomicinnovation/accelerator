# Research: Refine and consolidate the CHANGELOG for release 1.21.0

**Date**: 2026-06-04T11:40:26+00:00 (UTC)
**Author**: Toby Clemson
**Git Commit**: 36c9533859ad591d042c5942adbeff8abc88d5d4
**Branch**: main

## Research Question

We need to refine, simplify and clean up the `CHANGELOG.md` file for release
1.21.0, based on all changes since 1.20.0. This requires:

1. Consolidating the multiple copies of each heading section.
2. Checking that all changes are correctly represented in the changelog.
3. Ensuring nothing remains that is implementation detail (consumers of this
   file are end users of the plugin, not contributors).
4. Describing the requirement to run migrations when installing this version,
   with a brief description of the action of each migration.

This document establishes the authoritative final shipped state of every
feature area so a subsequent plan can rewrite the `[1.21.0]` section with
confidence.

## Summary

The `[1.21.0]` section (`CHANGELOG.md:5-411`) is an **unconsolidated
concatenation of prerelease changelog fragments**. It contains:

- **Three separate `### Added` blocks** (lines 7, 79, 196), plus single
  `### Breaking` (14), `### Changed` (118), `### Fixed` (178), and `### Notes`
  (340) blocks — these must be merged into one block per category.
- **A direct contradiction**: the Playwright MCP server / `.claude-plugin/.mcp.json`
  is described as **added** (273-274), **removed** (69-72), and **still active**
  (349-351) in the same release. Ground truth: it was **removed**; `.mcp.json`
  does not exist in the shipped tree and no `@playwright/mcp` reference survives
  in any skill or agent.
- **Stale intra-release path references**: the design-convergence block (255-274)
  describes intermediate state (Playwright MCP crawler; `paths.design_inventories`
  / `paths.design_gaps`) that was *superseded later in the same release cycle*
  by the `run.sh` executor and renamed config keys. The Jira block (363-411)
  references `meta/integrations/jira/*.json`, but the shipped default is
  `.accelerator/state/integrations/jira/`.
- **Substantial implementation detail** that does not belong in a user-facing
  changelog: a 50-line enumeration of internal Jira bash helper scripts
  (276-326), eval-id bookkeeping (73-77), internal SKILL.md preamble refactors
  (126-128, 188-190), the executor wire-protocol / `links` command internals
  (81-87, 114-116), the owner-PID daemon watcher internals (14-25, 191-194),
  and internal work-item references ("work item 0069", "work item 0052";
  lines 9, 89).
- **Two undocumented migrations**: the shipped `[1.21.0]` introduces migrations
  **0003, 0004, 0005, 0006**, but only 0003 and 0004 are described. **0005**
  (`type` → `kind`) and **0006** (`work_item_id` / `author` canonicalisation)
  are entirely absent, yet both rewrite frontmatter in the consumer's
  documents. There is no single place that tells an upgrader "run migrate; here
  is what each migration does".

**Verified delta since 1.20.0** (via `jj file list` at tags `1.20.0` and
`v1.21.0`): 2 new agents (`browser-locator`, `browser-analyser`); new skills
`research-issue`, `visualise`, `inventory-design`, `analyse-design-gaps`, the
8 Jira skills, and the `config/paths` + `config/browser-executor` preload
skills; new migrations 0003–0006 (1.20.0 shipped only 0001–0002).

## Detailed Findings

### 1. Structural problem: duplicated category headings

The `[1.21.0]` block repeats category headings because each prerelease appended
its own `### Added` / `### Changed` / etc. without consolidation:

| Heading | Line | Content |
|---|---|---|
| `### Added` | 7 | Migration framework interactive contract |
| `### Breaking` | 14 | owner-PID watcher removal; research restructure; Node≥20; `.mcp.json` removal; eval-id changes |
| `### Added` | 79 | `links` command; `browser-executor` skill; config warning; inventory-design flags/executor/bootstrap; PROTOCOL.md |
| `### Changed` | 118 | daemon idle timeout; SKILL.md preamble refactor; `.accelerator/` relocation (0003) details |
| `### Fixed` | 178 | browser-agent self-discovery; SPA route fabrication; config warning noise; daemon death |
| `### Added` | 196 | research-issue; visualiser; visualiser work-item terminology; design convergence; Jira |
| `### Notes` | 340 | inventory-design auth env vars; MCP pinning (stale); visualiser env vars; Jira skills catalogue |

`keepachangelog` standard categories are **Added / Changed / Deprecated /
Removed / Fixed / Security**. The file currently also uses non-standard
`Breaking`, `Notes` (and `Improved` in older sections). The release automation
(`tasks/changelog.py`, per `RELEASING.md:73,86`) only moves `[Unreleased]` to a
version heading — it does **not** normalise category names, so the structure
must be fixed by hand.

### 2. Playwright: MCP removed, executor is the final mechanism

Authoritative final state (agent-verified against the working tree):

- `.claude-plugin/.mcp.json` **does not exist** (the dir holds only
  `marketplace.json`, `marketplace-prerelease.json`, `plugin.json`). No
  `@playwright/mcp` or `mcp__playwright__*` reference remains in any skill or
  agent — the only matches are the contradictory CHANGELOG lines and tests that
  assert their *absence* (`scripts/test-design.sh:101-151`).
- `inventory-design` crawls via a **Bash-invoked Node.js executor**,
  `skills/design/inventory-design/scripts/playwright/run.sh`, which spawns a
  Playwright-Chromium TCP daemon (`lib/daemon.js:101,303`). Bootstrap is
  `scripts/ensure-playwright.sh`.
- Crawler modes `--crawler code | runtime | hybrid` (`SKILL.md:9,40-44`): `code`
  needs no Node; `runtime`/`hybrid` need **Node ≥ 20** (`ensure-playwright.sh:44`)
  and lazily install Chromium (~150 MB) into `~/.cache/accelerator/playwright/`;
  `hybrid` auto-downgrades to `code` on bootstrap failure, `runtime` hard-fails.
- Network-safety flags (`validate-source.sh`): `--allow-internal`
  (RFC1918/link-local/loopback-variant), `--allow-insecure-scheme` (plain http
  to non-localhost). `http://localhost` and `http://127.0.0.1` are default-allowed.
- Daemon idle default is **10 minutes** (`lib/daemon.js:21`, env
  `ACCELERATOR_PLAYWRIGHT_IDLE_MS`). The owner-PID watcher, `--owner-pid` arg,
  and `ACCELERATOR_PLAYWRIGHT_OWNER_POLL_MS` are **removed** (zero matches in
  `scripts/playwright/`). *Caveat for the editor*: the surviving `owner-pid`
  hits elsewhere belong to the unrelated visualiser launcher — do not conflate.
- `browser-locator` / `browser-analyser` drive the executor via
  `Bash(run.sh …)` (not MCP); the preloaded `accelerator:browser-executor`
  skill injects the resolved `run.sh` path (`skills/config/browser-executor/SKILL.md`).

**Implication for cleanup**: delete every "added `@playwright/mcp`" and "MCP
pinned in `.mcp.json`" statement; keep a single "Removed: Playwright MCP
integration, replaced by the `run.sh` executor" narrative. The `links` command
and PROTOCOL.md are internal executor wire-protocol details — drop from the
user changelog (or reduce to one neutral line).

### 6. The migration story — the core of requirement #4

`jj file list` at the two tags is definitive:

- **1.20.0** shipped migrations **0001, 0002** only.
- **v1.21.0** shipped **0001–0006**.

So a 1.20.0 → 1.21.0 upgrader runs **0003, 0004, 0005, 0006** via
`/accelerator:migrate`. **0005 and 0006 are completely absent from the
changelog.** Authoritative per-migration descriptions (agent-verified against
`skills/config/migrate/migrations/`):

- **0003 — Relocate Accelerator state into `.accelerator/`.** Moves team/personal
  config out of `.claude/`, and templates / integration state / migration
  tracking / tmp out of `meta/`, into the single `.accelerator/` tree (see §5).
  Also rewrites the root `.gitignore` Accelerator rule. Self-defers if there is
  nothing to move; aborts (with a `jj op restore` / `git reset` hint) on a
  source/destination conflict.
- **0004 — Restructure `meta/research/` into subcategories.** Existing research
  files move into `meta/research/codebase/`; a new `meta/research/issues/` is
  created; design-inventories/gaps nest under `meta/research/`. Renames config
  keys `paths.research` → `paths.research_codebase`, `paths.design_inventories`
  → `paths.research_design_inventories`, `paths.design_gaps` →
  `paths.research_design_gaps`, `templates.research` → `templates.codebase-research`,
  and adds `paths.research_issues`. Rewrites inbound path references across the
  configured corpora; backs up each touched config to `<cfg>.0004.bak`. Refuses
  to run without a VCS unless `ACCELERATOR_MIGRATE_FORCE_NO_VCS=1`.
- **0005 — Rename work-item `type` to `kind`.** In every doc under `paths.work`
  (default `meta/work/`), renames frontmatter `type:` → `kind:` and the body
  label `**Type**:` → `**Kind**:`. Idempotent; if a file already has both, it
  keeps `kind` and warns on value divergence.
- **0006 — Canonicalise `work_item_id` and `author`.** Across the plans,
  codebase-research, and issue-research corpora (and matching user-overridden
  templates): renames the legacy frontmatter key `work-item:` → `work_item_id:`
  and **quotes** its value (e.g. `work-item: 0034-foo` → `work_item_id: "0034-foo"`),
  and renames `researcher:` → `author:` (plus the `**Researcher**:` → `**Author**:`
  body label). Does not change ID values themselves. Keeps the canonical key and
  warns (`0006-DIVERGE`) when both legacy and canonical keys disagree; refuses
  unsafe values (`0006-REFUSE`).

**None of the six migrations is interactive** (no `# INTERACTIVE:` header in any
of them) — the interactive contract added in this release (work item 0069) is a
framework capability the runner dispatches on, but no shipped migration uses it.
The changelog's first `### Added` line conflates the *framework* contract with a
migration; keep the framework note, but don't imply any bundled migration prompts.

**Runner guarantees** worth a single consolidated mention: numeric ordering,
idempotency, refusal on a dirty tree, the pre-run banner with per-migration
preview, `ACCELERATOR_MIGRATE_FORCE=1` clean-tree bypass, VCS-revert recovery,
`--skip`/`--unskip`, the `MIGRATION_RESULT: no_op_pending` self-deferral
contract, and applied-tracking in `.accelerator/state/migrations-applied`.
README §Migrations (`README.md:111-140`) already states these well and can be
the reference target.

## Architecture Insights

- **The changelog is assembled by accretion, not curation.** Each prerelease
  appended a fragment; the release tool only relabels `[Unreleased]`. Within a
  single minor release, later prereleases can *supersede* earlier ones (MCP →
  executor; `design_*` keys → `research_design_*`), so consolidation is not
  mechanical de-duplication — it requires picking the **final shipped state** and
  discarding intermediate states that never reached a user.
- **Migrations are the user-visible contract of a release.** The four new
  migrations are the single most important thing for an upgrader, yet they are
  scattered (0003 buried in `### Changed`, 0004 in `### Breaking`, 0005/0006
  missing). A dedicated, ordered "Migrations" subsection is the highest-value
  structural change.
- **Audience boundary**: the file conflates two audiences. Helper-script
  inventories, eval ids, preamble refactors, wire protocols, and daemon
  internals are contributor concerns; they belong in commit history / research
  docs, not the consumer changelog.

## Open Questions

1. **Category convention**: adopt strict Keep-a-Changelog categories (Added /
   Changed / Removed / Fixed), folding the non-standard `Breaking` items into
   the relevant category with a **Breaking:** prefix — or keep a dedicated
   `### Breaking changes` callout (high user value, non-standard)? Confirm
   `keepachangelog`/`tasks/changelog.py` tolerates a custom `### Migrations`
   subsection (it appears to only relabel `[Unreleased]`, so yes).
2. **Migration placement**: dedicated `### Migrations` subsection vs. inline
   `(migration 000N)` tags on each Changed entry vs. both.
3. **Node ≥ 20 requirement**: how prominent? It only affects
   `inventory-design --crawler runtime|hybrid`; `plugin.json` already lists it
   under `requirements`.
4. **Forward behaviour entries (§7)**: add `### Changed` lines for the
   `type`→`kind` and `author`/`work_item_id` skill/template changes, or let the
   migration descriptions carry it?
5. **Historical entries**: leave the existing note at `CHANGELOG.md:59-61`
   (older entries reference legacy paths verbatim) — confirm we are *not*
   rewriting pre-1.21.0 sections, only consolidating 1.21.0.
