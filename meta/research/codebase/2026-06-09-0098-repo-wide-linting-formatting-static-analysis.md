# Research: Codebase context for repo-wide linting, formatting, and static analysis guardrails (0098)

**Date**: 2026-06-09T21:17:04+00:00 (UTC)
**Author**: Toby Clemson
**Git Commit**: 7d2dd42793f640752ec85594a8775f7a1163c6db
**Branch**: detached HEAD (jj-colocated working copy on top of main)
**Repository**: accelerator

## Research Question

Gather full codebase context for implementing work item 0098 — repo-wide
linting, formatting, and static-analysis guardrails covering the visualiser
frontend (TypeScript/React, Biome + `tsc --noEmit`), the visualiser server
(Rust, rustfmt + clippy), Python (ruff + pyrefly), and shell
(shellcheck + shfmt), wired into mise tasks and Main CI as blocking
per-language jobs.

## Summary

The repository is well-positioned for this work, and two of the work item's
premises need material correction:

1. **Python is ~10× smaller than assumed.** There are exactly **63 tracked
   `.py` files** (37 in `tasks/`, 25 in `tests/`, 1 in `skills/`), not
   "~700+". The larger figure counts `.venv/` site-packages (and the
   per-workspace venvs). The Python remediation sweep is small, the code is
   already consistently typed (modern 3.14-era syntax), and only 7 `# noqa`
   markers exist — one already using a ruff code (`SIM115`).

2. **Shell tooling is already half-built.** shellcheck (0.11.0) and shfmt
   (3.13.1) are already pinned in `mise.toml`, already run in CI via the
   `check` job (`mise run check` → `format:scripts:check` +
   `lint:scripts:shellcheck:check` + `lint:scripts:bashisms:check`), and
   shfmt already sources style from `.editorconfig` with no CLI flags. The
   delta for 0098 is: a repo-root `.shellcheckrc` with `enable=all`
   (currently the gate is `-x --severity=warning` with default checks), a
   scope decision about 12 currently-excluded files, and one extensionless
   bash script outside every glob.

Other load-bearing findings:

- **Task-naming conflict**: the work item mandates `lint:frontend` /
  `lint:rust` / `lint:python` / `lint:shell` + aggregate `lint`, but the
  repo's established convention is `lint:scripts:shellcheck:check`-style
  leaves aggregated under `lint:check` / `format:check` / `check` ("what CI
  runs"), with implementations delegated to Python invoke tasks. Planning
  must reconcile the two shapes.
- **`Cargo.toml` is regenerated by version tooling** (`tasks/version.py`) —
  a new `[lints.clippy]` table must survive that templating or it will be
  silently dropped on the next version bump.
- **The frontend `typecheck` script is currently a no-op**: `tsc --noEmit`
  runs against the solution-style `tsconfig.json` whose `files` is `[]`;
  plain-mode tsc ignores `references`, so nothing is checked. Several files
  (Playwright configs, `scripts/scan-css-literals.ts`, `e2e/start-server.mjs`)
  are in no tsconfig at all.
- **Biome defaults would reformat the entire frontend** — the prevailing
  style is single quotes / no semicolons / 2-space indent; Biome defaults to
  double quotes and semicolons.
- **Pin-policy gaps already exist**: `node = "22"` (major-only) in
  `mise.toml`, `typescript: ^5` (caret) in `package.json`, and
  `jdx/mise-action@v4` (tag, not exact) in workflows.
- **CSS assumption confirmed**: all frontend CSS is plain CSS (40
  `*.module.css` + 3 globals); no SCSS/PostCSS/Tailwind anywhere. The only
  CSS outside the frontend is two design-prototype artifacts under
  `meta/research/design-inventories/` (out of scope per the work item).

## Detailed Findings

### Frontend (TypeScript/React) — `skills/visualisation/visualise/frontend/`

**Toolchain today**: no Biome, ESLint, Prettier, or stylelint anywhere — no
lint/format config files and no lint/format npm scripts. The only quality
gates are `tsc -b` inside `npm run build` and the Vitest suite.

**package.json** (`frontend/package.json`):
- `type: "module"`, `engines.node >=22`, no `packageManager` field; npm with
  `package-lock.json` (lockfileVersion 3, tracked; installs via
  `npm --prefix … ci` from `tasks/deps.py:24`).
- Version pinning is mixed: exact pins (`@dnd-kit/*`) alongside major-only
  carets (`react ^19`, `typescript ^5` at line 47, `vite ^6`, `vitest ^3`).
  The pin AC requires biome and typescript at exact versions — typescript's
  caret must be tightened.
- Scripts: `typecheck: "tsc --noEmit"` exists (line 12) but is invoked by
  nothing in mise/CI — and is a **no-op** (see tsconfig below). `build` runs
  `tsc -b && vite build`.

**TypeScript configs** (solution-style, 4 files):
- `tsconfig.json` — pure solution file: `"files": []` + references to
  `tsconfig.app.json` / `tsconfig.node.json` / `tsconfig.e2e.json`. Plain
  `tsc --noEmit` against it checks zero files; the standalone type-check
  must use `tsc -b --noEmit` (or equivalent) to honour the references.
- `tsconfig.app.json` — `strict: true`, `noUnusedLocals`,
  `noUnusedParameters`, `noFallthroughCasesInSwitch`; includes `src` only;
  no path aliases anywhere (also no `resolve.alias` in vite.config.ts).
- `tsconfig.node.json` — covers only `vite.config.ts`; **strict is off**.
- `tsconfig.e2e.json` — covers `e2e/` + `tests/`; strict on; contains a
  **duplicate `"types"` key** (lines 7 and 13) that Biome's
  `noDuplicateObjectKeys` will flag the moment JSON linting turns on.
- **Type-check coverage gaps**: `playwright.config.ts`,
  `playwright.glyph.config.ts`, `scripts/scan-css-literals.ts`, and
  `e2e/start-server.mjs` are in no tsconfig's include set — never checked.

**Source shape**: 237 `.ts`/`.tsx` under `src/` (106 co-located
`*.test.ts(x)`), 20 files in `e2e/` (Playwright `*.spec.ts` + 1 plain-JS
`start-server.mjs`), 24 in `tests/visual-regression/` (`*.spec.ts`), 3
root-level TS configs. Vitest/Playwright conventions split cleanly
(`*.test.*` vs `*.spec.*`).

**CSS**: 43 files — 40 `*.module.css` co-located with components, 3 globals
(`src/styles/global.css`, `code-syntax.global.css`, `wiki-links.global.css`).
No SCSS/Sass/Less, no PostCSS/Tailwind config. CSS hygiene is currently
enforced through Vitest (`src/styles/migration.test.ts` +
`scripts/scan-css-literals.ts`), not a linter — these test-based gates
(ADR-0036, ADR-0039) remain authoritative and are not displaced by Biome.

**Prevailing format style** (vite.config.ts, playwright.config.ts,
src/test/setup.ts): **single quotes, no semicolons, trailing commas,
2-space indent**. Biome defaults (double quotes, semicolons) would rewrite
the whole codebase — config needs `quoteStyle: "single"`,
`semicolons: "asNeeded"` (or accept a full-reformat sweep, which conflicts
with keeping diffs reviewable).

**Existing suppressions** (4 total):
- `src/components/Glyph/Glyph.test.tsx:13` — the only `@ts-expect-error`,
  documented intentional compile-time guard.
- `src/components/Sidebar/SearchResultsPanel.test.tsx:36,38` and
  `src/api/boot-theme.test.ts:121` — three **vestigial `eslint-disable`
  comments** (no ESLint installed); candidates for deletion in the sweep.
- Zero `@ts-ignore`, `@ts-nocheck`, `biome-ignore`.

**Files Biome must exclude**: `node_modules/`, `dist/` (gitignored at repo
root only, not in the frontend `.gitignore`),
`tests/visual-regression/__screenshots__/` (committed PNG baselines),

### Rust server — `skills/visualisation/visualise/server/`

**Lint state today**: no `[lints]` tables in `Cargo.toml`, no
`rustfmt.toml`/`clippy.toml` anywhere, no fmt/clippy task or CI step. The
`[lints.clippy]` table is entirely net-new.

**Crate shape**: single crate (no workspace; the only real Cargo.toml in the
repo), lib + bin, edition 2021, `rust-version = "1.85"` (MSRV; comment at
`Cargo.toml:5-7` explains the mise toolchain is newer because dev-deps need
1.86+). 58 `.rs` files: 38 under `src/` (23 public modules + 13 `src/api/`
modules), 19 under `tests/` (18 integration roots + `tests/common/mod.rs`),
plus a 25-line `build.rs`. `Cargo.lock` is tracked (v4).

**Toolchain pin**: solely `mise.toml:8` — `rust = "1.90.0"` (exact). No
rust-toolchain.toml. This satisfies the pin AC as-is.

**⚠ Templating landmine**: `tasks/version.py:31,57,61`
(`_render_cargo_toml` / `atomic_write_text(CARGO_TOML, …)`) regenerates
Cargo.toml on version bumps. A `[lints.clippy]` table added to Cargo.toml
must survive this rendering — verify/extend the template, or the table
vanishes on the next release.

**⚠ Feature-matrix coverage**: features are `default = ["embed-dist"]` and
`dev-frontend`, with mutually exclusive `cfg` arms in `src/assets.rs`
(lines 21-27, 33, 53, 65, 70, 75, 83, 157, 210). `--all-features` enables
`dev-frontend`, which compiles **out** the `cfg(not(feature =
"dev-frontend"))` arms — so a single
`cargo clippy --all-targets --all-features` run does not see all code.
Additionally `build.rs` panics if `frontend/dist/index.html` is missing
whenever `embed-dist` is on — so any clippy invocation that enables
`embed-dist` (including `--all-features`) **requires the frontend to be
built first** (mise `depends` on `build:frontend`, mirroring
`test:unit:visualiser` at `mise.toml:96`). Precedent for two-pass coverage:
`tasks/test/unit.py:6-24` runs `cargo test --lib` twice, once per feature
config.

**Existing suppressions** (5 total, no crate-level `#![allow]`):
- `src/watcher.rs:32,110,354` — `#[allow(clippy::too_many_arguments)]` ×3.
- `tests/common/mod.rs:6,89` — `#[allow(dead_code)]` ×2.

**unsafe**: 4 FFI blocks (`src/main.rs:65-66` with a `// SAFETY:` comment;
`src/server.rs:502,536-545` without — only relevant if restriction-group
lints are considered; pedantic doesn't hit them).

**pyproject.toml** (39 lines): `requires-python = ">=3.14"`; no

(`build` and `dev`); `[tool.uv] prerelease = "allow"`;
`[tool.setuptools] packages = []`. **No ruff/mypy/pyrefly/pyright/black
config anywhere** — `[tool.ruff]` and `[tool.pyrefly]` are net-new.
`uv.lock` exists; deps install via the mise postinstall hook
(`uv sync --only-group build --frozen`, `mise.toml:19`). Python pinned at
`3.14.4` (`mise.toml:6`), uv at `0.11.6` (`mise.toml:5`).

**Type-hint culture**: `tasks/` production code is consistently typed with
modern syntax (`X | None`, builtin generics, `Protocol`, frozen dataclasses,
`StrEnum`); no `from __future__ import annotations` anywhere (relies on the
3.14 floor). Known gaps pyrefly-strict will surface: `@task` functions never
annotate returns (`tasks/build.py:104`), occasional untyped params
(`tasks/build.py:57`), and tests have partially-annotated fixtures.
`mock-jira-server.py` reads ad-hoc attributes off `self.server`
(lines 31-32, 54) — pyrefly strict will flag unknown attributes on
`HTTPServer`.

**Per-file complications**:
- `mock-jira-server.py` self-guards for **Python 3.9+** (lines 4-8) because

  ruff `target-version`/pyrefly `python-version` may need per-directory

  standalone unimportable file).
- Third-party imports in tracked code: invoke, pytest, psutil, semver,
  tomlkit, keepachangelog, pytest-mock (via fixture). All declared in
  dependency groups, so pyrefly can resolve them from the venv.

**Existing suppressions** (7 `# noqa`, zero `# type: ignore`):
`tasks/shared/dev/circus.py:185` (`SIM115`, with rationale — ruff codes
already informally in use), `tests/integration/dev/dev_integration_driver.py:26-40`
(`E402` ×5), `tests/unit/tasks/test_dev.py:22` (`F401`).

**Excludes needed**: `.venv/` is **not** in `.gitignore` (covered only by a
machine-global ignore) — ruff's defaults exclude it but an explicit
`exclude` entry is the safe move; `workspaces/` is gitignored but exists on
disk with full repo copies + venvs.

**Delta to reach 0098's target state**:

`#!/usr/bin/env bash`; 2-space indent everywhere (zero tab-indented lines);
`set -euo pipefail` in 140 files with documented conventions for the
exceptions (sourced libraries inherit caller options per
`scripts/config-common.sh:4-5`; hooks and trivial wrappers omit it);
shfmt-conformant style already gated in CI.

**Existing directives** (72): 35 `disable=` across 11 codes (SC2034 ×8,
SC2059 ×8, SC2206 ×6, SC2069 ×3, SC1091 ×3, SC1090 ×2 file-level, plus
SC2016/SC2020/SC2086/SC2154/SC2329 ×1), 36 `source=` (8 of them
`source=/dev/null` in test harnesses), 1 redundant `shell=bash`. The
suppression-comment AC will require auditing these for adjacent rationale.

`skills/config/migrate/scripts/interactive-lib.sh`) — the current `-x`
invocation from repo root plus the 36 `source=` directives make these
resolve today; `.shellcheckrc` must preserve equivalent behaviour
(`external-sources=true` is not honoured from rc files by all versions —
shellcheck 0.11.0 does support it in rc; verify during implementation).

**Bash 3.2 floor (ADR-0016)**: enforced by `scripts/lint-bashisms.sh`
(denies `declare -A`, `mapfile`, `${var,,}`, `|&`, etc.; per-line opt-out

the bashisms lint remains a sibling check, not replaced by 0098 tooling.
One declared exception: `hooks/vcs-guard.sh:7` requires bash 4+.

**.editorconfig** (repo root, 13 lines): `root = true`; `[*.tsv]` and

`switch_case_indent = true`, with a comment documenting shfmt sourcing). No
`max_line_length`, no `[*]` defaults — the ruff/rustfmt cross-reference AC
needs values to reference, so `.editorconfig` may need extending (or the
cross-reference comments reference the absence/defaults).

### mise tasks and tool pins — `mise.toml`

**Pins** (`mise.toml:4-13`): all exact except node — `uv 0.11.6`,
`python 3.14.4`, `gh 2.89.0`, `rust 1.90.0`, **`node "22"` (major-only)**,
`shellcheck 0.11.0`, `shfmt 3.13.1`, `jj 0.36.0`, `jq 1.7.1`. ruff and
pyrefly are net-new pins. `[settings] experimental = true`;
`[hooks] postinstall = "uv sync --only-group build --frozen"` is what makes
invoke-backed tasks work immediately after `jdx/mise-action`.

**Task conventions**: colon-separated `<verb>:<scope>:<subscope>` (e.g.
`test:unit:frontend`, `build:server:dev`); check/fix leaves end `:check` /
`:fix` (`format:scripts:check`, `lint:scripts:shellcheck:check`); aggregates
are pure `depends` fan-outs with no `run` (`test:unit`, `lint:check`,
`format:check`, `check` at `mise.toml:216-218` — "what CI runs"); **every
task has a `description`**; implementations delegate to Python invoke
(`invoke lint.scripts.shellcheck` etc., namespace in `tasks/__init__.py`),
except `test:unit:tasks`/`test:integration:tasks` which run
`uv run pytest …` directly.

**⚠ Naming reconciliation needed**: the work item's Entry-point AC mandates
`lint:frontend` / `lint:rust` / `lint:python` / `lint:shell` + aggregate
`lint`, but the existing shape would suggest e.g. `lint:scripts:*` under
`lint:check` and format leaves under `format:check`, with `check` as the CI
aggregate. Options at planning time: adopt the work item names as new
aggregates that fan out to convention-shaped leaves; or fold the new checks
into the existing `lint:check`/`format:check`/`check` tree and treat the AC
naming as satisfied by per-language aggregates. Also note the existing tree
separates `format:*` from `lint:*`, while the work item folds format checks
into `lint:<language>`.

### CI — `.github/workflows/`

(environment-gated). **No `continue-on-error`, no `concurrency`, no
`timeout-minutes` anywhere.** Every job runs exactly one mise task — the
work item's "CI jobs execute the mise tasks" requirement matches existing
practice exactly. New per-language lint jobs follow the same three-step
shape.

**Pin status**: `jdx/mise-action@v4`, `actions/checkout@v5`,
`actions/attest-build-provenance@v2` are all tag pins — the work item
requires mise-action pinned "consistent with the exact-pin policy" (i.e.
exact version tag or SHA).

**npm/cargo/uv setup in CI**: never inline — flows through mise `depends`
chains (`deps:install:node` → `npm --prefix … ci`) and the postinstall
hook. Only mise tool installs are cached (`cache: true`); no
actions/cache for npm, cargo, or uv — a clippy job will cold-compile unless
caching is added (optional, not in scope).

**0090 gate clarification**: the work item describes 0090's gate as a
"standing radius-token CI grep gate", but as shipped it is a **Vitest
suite** — `src/styles/migration.test.ts:788-818`
(`BORDER_RADIUS_LITERAL_RE`, declared the authoritative implementation of
ADR-0039 at lines 779-786), running under `test:unit:frontend` in the
`test-unit` job. There is no workflow-level grep step. Consolidation into
the lint structure remains explicitly discretionary per the work item.

**Branch protection**: no settings files or docs declare required checks;
gating is `needs:` + the `release` environment. The work item's assumption
that "Main CI's status is already required for merge" is not verifiable
from the repo — it lives in GitHub settings.

## Architecture Insights

1. **"CI runs one mise task per job" is a hard convention.** Every Main CI
   job is checkout → mise-action → `mise run <aggregate>`; tasks delegate to
   Python invoke for anything non-trivial, and source-set enumeration is
   centralised (`tasks/shared/sources.py`). New lint tooling that follows
   this shape (invoke implementations + mise leaves + aggregate fan-out +
   thin CI jobs) will fit; inline workflow commands would be the anomaly.
2. **Quality gates already live in two registers**: tool-based checks under
   `mise run check`, and convention-enforcement tests inside the unit suites
   (design-token gates in `migration.test.ts`, template/frontmatter checks
   in `test:unit:templates`). 0098 adds to the first register; the second is
   untouched (and the bashisms lint sits in the first already).
3. **Single-source-of-style is deliberate**: `.editorconfig` drives shfmt
   precisely because no CLI flags are passed (documented in
   `tasks/format/scripts.py`); the same philosophy argues for Biome's native
   `.editorconfig` support and explicit cross-reference comments in
   ruff/rustfmt config.
4. **The repo's Python is infrastructure code, not product code** — invoke
   tasks and their tests. It is already typed to a high standard, so pyrefly
   strict is plausible with modest remediation; the single 3.9-floor script
   in `skills/` is the only versioning wrinkle.
5. **Version templating touches Cargo.toml and is the one place where
   config-as-code gets regenerated** — any new committed config in that file
   must be reflected in `tasks/version.py`'s rendering.
6. **Feature flags fragment Rust lint coverage** the same way they fragment
   test coverage — and the repo already solved this for tests by running
   twice. Mirroring that pattern for clippy keeps coverage honest.

## Open Questions

1. **Task naming**: adopt the work item's `lint:<language>` + `lint` names,
   or extend the existing `lint:check`/`format:check`/`check` tree (which
   separates format from lint and ends leaves in `:check`)? The AC is
   phrased against the former; the codebase convention is the latter. Also:
   does the existing `check` aggregate absorb the new per-language tasks
   (one CI job) or do the four new per-language CI jobs replace/sit beside
   the `check` job?
2. **Shell scope widening**: do the 9 `test-fixtures` `.sh` files and 3
   `test-helpers.sh` files enter scope (the coverage AC implies yes, with
   justification comments required for any continued exclusion)? Does the
   extensionless `cli/accelerator-visualiser` get added explicitly?
3. **Frontend format style**: configure Biome to match the existing
   single-quote/no-semicolon style (minimal sweep diff) or accept Biome
   defaults (full reformat)? The work item is silent on formatter style.
4. **`tsc --noEmit` mechanics**: the standalone type-check must be
   `tsc -b --noEmit` (or per-project `-p` invocations) to check anything;
   should the uncovered files (Playwright configs, scripts, start-server.mjs)
   be brought into a tsconfig as part of this item?
5. **Clippy feature matrix**: one `--all-features` run (per the AC's literal
   command, missing the `not(dev-frontend)` arms) or two runs mirroring the
   unit-test pattern? Either way `build:frontend` must precede any
   `embed-dist`-enabled invocation.
6. **`.editorconfig` extension**: ruff/rustfmt cross-reference comments need
   concrete line-length/indent values — extend `.editorconfig` with TS/
   Python/Rust sections (and a `max_line_length`), or document that the
   tools' defaults are the reference?
7. **Cargo.toml templating**: confirm `tasks/version.py` preserves unknown
   tables when rendering — if it re-renders from a template rather than
   patching in place, the `[lints.clippy]` table needs adding there too.
8. **Pin tightening side-quests**: does this item also tighten `node = "22"`
   and the `jdx/mise-action@v4` tag (the AC names mise-action; node is
   arguably out of scope but inconsistent)?
