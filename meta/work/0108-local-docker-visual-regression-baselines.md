# 0108: Local Docker-Based Visual Regression Baseline Generation

**Kind**: Story
**Status**: Ready
**Priority**: Medium
**Author**: Toby Clemson

## Summary

As a developer changing the frontend, I want to regenerate the visual
regression baseline snapshots locally with a single task, so that I can
commit the refreshed baselines alongside my change and trust the CI
pipeline will pass on the first run.

Today, Linux baselines are regenerated by a separate GitHub Actions
workflow that pushes the new snapshots back to `main`. Any frontend
change therefore fails its own pipeline, and there is no way to re-run
that original pipeline against the regenerated baselines because the
push creates a new commit. This work moves baseline generation to the
local development loop and removes the CI push-back entirely.

## Context

The repository currently commits per-platform baselines
(`-darwin.png` and `-linux.png`) under
`skills/visualisation/visualise/frontend/tests/visual-regression/__screenshots__/`
(convention established in work item 0037). Linux baselines lag behind
darwin because they can only be produced in CI, and the "Update visual
regression baselines" GitHub Actions (GHA) workflow regenerates them by
pushing back to `main` — which also will not re-trigger Main CI under
`GITHUB_TOKEN`.

Playwright snapshots differ between macOS and Linux by design: the two
platforms use different text rasterizers (CoreText/Quartz vs
FreeType/HarfBuzz) and antialiasing, producing pixel deltas far too
large to absorb with tolerance settings. A single baseline cannot be
shared across two *host* environments — but it can be shared if the
*environment itself* is made singular. Generating all baselines inside
the official Playwright Linux Docker image
(`mcr.microsoft.com/playwright`), pinned to the project's Playwright
version, yields one canonical baseline set that every developer and CI
run compares against. This is the industry-standard replacement for a
CI-push-back workflow (see the Playwright Docker reference below).

The internal team works exclusively on macOS; public contributions may
arrive from Linux. The chosen approach (single Linux baseline via
Docker) means visual assertions become "Docker-only": native macOS test
runs will no longer self-validate against the committed baselines, and
the visual specs must be isolated so a native run does not fail on them.

## Requirements

- Provide a `mise`/`invoke` task that runs the visual-regression specs
  with `--update-snapshots` inside the pinned official Playwright Docker
  image, regenerating the committed baseline set.
- Provide a way (the same task in compare mode, or a companion task) to
  *run* the visual specs in Docker locally, so a developer can verify a
  clean pass before committing.
- Pin the Docker image tag to the project's Playwright version via a
  single shared source of truth, referenced by both the local task and
  the CI workflow, so they cannot drift.
- Pin the Chromium headless build (`channel`) and locale
  (`LANG`/`LC_ALL`) consistently across the local Docker task and CI.
- Pin the Docker image *platform* to `linux/amd64` (matching CI) on both
  the local task and CI, so Apple Silicon contributors run the amd64 image
  under emulation rather than pulling the `arm64` variant — the
  single-baseline-set promise only holds when every run renders on the
  same CPU architecture.
- Isolate the visual-regression specs (separate Playwright project or
  tag) so a native local test run does not execute or fail on the
  Docker-generated baselines.
- Run the visual-regression specs in CI as a dedicated job that executes
  only on Linux, in the pinned Playwright Docker image — removing them
  from the macOS matrix leg of the existing `test-e2e` job entirely (so
  the macOS leg never needs a Docker daemon).
- Remove the existing "Update visual regression baselines" GHA workflow.
- Remove the existing `-darwin.png` baselines and the `{platform}` token
  from `snapshotPathTemplate`, leaving a single Linux baseline set.
- Document the new contributor workflow (including for Linux/public
  contributors): when and how to re-baseline, the Docker requirement,
  and how to debug visual tests locally.

## Acceptance Criteria

- [ ] Given a developer has changed the frontend, when they run the
      re-baseline task and commit the regenerated baselines and push the
      branch, then the visual-regression CI job on that same commit
      passes with zero pixel diffs against the committed baselines.
- [ ] Given any contributor (including on Apple Silicon) using the pinned
      image (which bundles its own fonts), tag, `linux/amd64` platform,
      headless build, and locale, when they run the local Docker compare
      task against the committed baselines, then it passes with zero pixel
      diffs — confirming their environment reproduces the CI-generated
      reference set regardless of host architecture.
- [ ] Given the repository after this change, when
      `skills/visualisation/visualise/frontend/tests/visual-regression/__screenshots__/`
      is inspected, then only one baseline set exists (no
      `-darwin`/`-linux` split).
- [ ] Given the change is merged, when the GitHub Actions workflows are
      inspected, then the "Update visual regression baselines" workflow
      no longer exists and no mechanism pushes regenerated baselines
      back to `main`.
- [ ] Given a clean checkout of `main` whose baselines were generated by
      CI (the committed reference set), when a developer runs the local
      Docker compare task without re-baselining, then it passes with zero
      diffs — confirming local Docker output matches the committed
      baselines.
- [ ] Given a developer runs the project's standard (non-visual) test
      tasks natively on macOS (e.g. `mise run test` and
      `mise run test:e2e`), when those run, then the visual-regression
      specs are skipped rather than failing against Docker baselines.
- [ ] Given the CI pipeline after this change, when the `test-e2e`
      matrix and the visual-regression job are inspected, then the visual
      specs run only in the dedicated Linux Docker job and do not run on
      the macOS matrix leg.
- [ ] Given the Playwright version is bumped in `package.json`, when both
      the local Docker task and the CI visual job run, then both resolve
      to the same Docker image tag and Chromium `channel` from the single
      shared source of truth without manual edits to either.
- [ ] Given the pinned locale (`LANG`/`LC_ALL`), when both the local
      Docker task and the CI visual job run, then both read the identical
      locale value from the same shared source — confirming, together
      with the tag/channel criterion above, the no-drift guarantee across
      every value that must stay aligned.
- [ ] Given the visual-regression config after this change, when the
      per-assertion comparison tolerances are inspected, then
      `maxDiffPixelRatio` remains at or below the current per-spec values
      (0.05 / 0.01) and no global `threshold` raise is introduced — so
      tolerances absorb only residual in-environment flake, never a
      cross-environment delta.
- [ ] Given the change is merged, when the contributor/testing
      documentation is inspected, then it covers the Docker prerequisite,
      when and how to re-baseline, and how to debug visual tests locally.

## Dependencies

- Blocked by: no internal work items. Two external/tooling couplings the
  approach hard-depends on (see below) are not blockers but must be acknowledged.
- External — Microsoft Container Registry (`mcr.microsoft.com/playwright`):
  every re-baseline run, local compare run, and CI visual job pulls the official
  image, so the work depends on MCR being reachable and on Microsoft continuing
  to publish a tag matching the resolved `@playwright/test` version. There is no
  in-repo fallback if a tag for a future pinned version is unavailable. Both
  risks — MCR unreachability and a missing tag for a future pinned version — are
  accepted, not mitigated.
- Tooling — Docker daemon: a usable Docker daemon is a hard prerequisite for all
  visual-test work (running and re-baselining), for both contributors and the CI
  visual-regression job. That job runs only on Linux (`ubuntu-latest`), where the
  GitHub-hosted runner provides a Docker daemon; the macOS matrix leg no longer
  runs visual specs, so it needs no daemon. Local visual work on macOS requires a
  developer-provided Docker daemon (e.g. Docker Desktop).
- Blocks: 0082 — BigGlyph Hero Illustration Set (status: ready). Soft
  coordination edge, not a hard technical block: 0082 will capture new visual
  baselines. If 0108 lands first, those baselines are generated under the
  Docker single-Linux regime; otherwise 0082's dual-platform baselines need
  regenerating once 0108 collapses the set. Sequencing 0108 ahead avoids the
  rework. 0082 is the only outstanding (non-`done`) work item that captures
  visual-regression baselines; 0037, 0073, and 0095 are all `done` and so carry
  no live coupling to the baseline-set collapse.

## Assumptions

- Docker as a hard dependency for any visual-test work (running or
  re-baselining) is acceptable for all contributors, internal and
  public.
- Dropping the `-darwin` baselines entirely is acceptable given the goal
  of eliminating platform-specific baselines.
- The internal (all-macOS) team accepts that native-macOS visual runs no
  longer self-validate, and that Docker becomes the sole source of truth
  for visual-test passes.

## Technical Notes

**Size**: L — Greenfield Docker infrastructure (no `Dockerfile`/compose exists
anywhere in the repo per the codebase scan), plus: new re-baseline + compare
mise/invoke tasks wired alongside `test:e2e:visualiser` (`mise.toml:172`,
`tasks/test/e2e.py`); a version-pin wrapper deriving the image tag from the
*resolved* `@playwright/test ^1.59.1` (`package.json:39`); adding an explicit
`snapshotPathTemplate` and extracting the visual specs into a dedicated
Linux-only CI job (out of the `test-e2e` matrix) plus isolating them from native
local runs (`playwright.config.ts:24-36`, `main.yml:59-79`); regenerating +
collapsing the baseline set across 23 specs; removing
`update-visual-baselines.yml`; and contributor docs. Breadth across config,
tasks, CI, and docs — not just volume — drives the L.

- Use the official `mcr.microsoft.com/playwright:vX.Y.Z-<flavour>` image,
  matching the Ubuntu flavour CI uses (`-noble` / `-jammy`), tag pinned
  to the project's Playwright version.
- Run the container with `--ipc=host` to avoid Chromium out-of-memory
  (OOM) crashes on large screenshot runs.
- Run the container with `--platform=linux/amd64` so Apple Silicon hosts
  emulate the amd64 image instead of pulling the `arm64` variant; without
  this, Chromium/Skia rendering and PNG encoding differ at the byte level
  across architectures and the single-baseline-set promise breaks. Accept
  the emulation overhead on M-series Macs as the cost of cross-architecture
  reproducibility.
- Drop the `{platform}` token from `snapshotPathTemplate` so a single
  filename is shared; this is only safe because rendering is now always
  the Docker environment.
- Keep `threshold` / `maxDiffPixelRatio` modest — for residual
  in-environment flake only, never as a cross-platform bridge.
- A thin wrapper invoked by the mise task can auto-detect the installed
  Playwright version to keep the image tag aligned with `package.json`.
- Consider (separately, out of scope) Git LFS if the committed PNG set
  grows large.
- Cutover ordering (single atomic increment): introduce the shared
  version-pin source, the local Docker task, and the CI workflow change
  together so local and CI never reference divergent image tags mid-change;
  keep `update-visual-baselines.yml` in place until the new Docker task is
  verified passing in CI, so neither the old nor the new path leaves the
  suite broken during the transition.

### Codebase anchors (added during refinement)

- **Specs and baselines live in the frontend package, not a top-level dir.**
  Specs: `skills/visualisation/visualise/frontend/tests/visual-regression/`
  (23 `.spec.ts` files). Baselines:
  `.../tests/visual-regression/__screenshots__/<spec>.spec.ts-snapshots/`
  with `…-visual-regression-{darwin,linux}.png` naming. The Summary, Context,
  and Acceptance Criteria use this fully-qualified path throughout.
- **Spec isolation is already partly done.** `playwright.config.ts:24-36`
  defines a `visual-regression` project (`testDir: ./tests/visual-regression`,
  `snapshotDir: ./tests/visual-regression/__screenshots__`) and a dependent
  `chromium` project (`dependencies: ["visual-regression"]`). The isolation
  requirement is about ensuring native macOS runs *skip* the visual project,
  not creating the project from scratch.
- **`snapshotPathTemplate` is not set** — the `{platform}` suffix comes from
  Playwright's default template. Dropping the token therefore means *adding*
  an explicit `snapshotPathTemplate` without `{platform}`, not editing an
  existing one (`playwright.config.ts`, no `expect`/template block today).
- **`maxDiffPixelRatio` is per-assertion, not global**: `0.05` across most
  showcase specs (e.g. `tokens.spec.ts:22`), `0.01` in
  `task-list-visual.spec.ts:19`. No `threshold` is set anywhere. There is no
  global `expect` block to tune — keep per-assertion or introduce a global one.
- **Push-back workflow to remove**: `.github/workflows/update-visual-baselines.yml`
  — `workflow_dispatch`-only, `ubuntu-latest`, runs
  `npx playwright test --project visual-regression --update-snapshots`
  (lines 32-36) then commits + `git push origin HEAD:<ref>` under
  `GITHUB_TOKEN` (lines 38-44), which is why it cannot re-trigger Main CI.
- **CI runs visual specs on a matrix**: `.github/workflows/main.yml:59-79`,
  `test-e2e` job, `os: [ubuntu-latest, macos-latest]`, via `mise run test:e2e`.
  Collapsing to a single Linux baseline means extracting the visual-regression
  specs into a dedicated CI job that runs only on Linux (`ubuntu-latest`) in the
  pinned Docker image, and dropping them from the `test-e2e` matrix so the macOS
  leg never runs (or needs a Docker daemon for) the visual project. Ties to the
  dedicated-Linux-job and native-macOS-skip acceptance criteria.
- **Task wiring**: `mise.toml` — `test:e2e:visualiser` (172-175) →
  `invoke test.e2e.visualiser`; `test:e2e` (177), `test` (181);
  `deps:install:playwright` (33-36). Invoke bodies: `tasks/test/e2e.py:19-22`
  (`npm --prefix <fe> run test:e2e`), `tasks/deps.py:30-32`
  (`playwright install --with-deps chromium`). npm script:
  `package.json:21` (`"test:e2e": "playwright test"`). New Docker re-baseline /
  compare tasks slot in alongside these.
- **Version source of truth**: `package.json:39` —
  `"@playwright/test": "^1.59.1"` (caret range, not exactly pinned). The image
  tag must derive from the *resolved* installed version, not the caret literal.
- **No locale or Chromium `channel` is set today** (`playwright.config.ts`
  only sets `browserName: "chromium"`); the server is spawned inheriting
  `process.env` in `e2e/start-server.mjs`. `LANG`/`LC_ALL`/`channel` pinning
  is net-new.
- **Greenfield Docker**: no `Dockerfile`, `docker-compose.yml`, or
  `mcr.microsoft.com/playwright` reference exists anywhere in the repo. This
  is entirely new infrastructure.
