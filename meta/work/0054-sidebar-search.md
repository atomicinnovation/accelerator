# 0054: Sidebar Search Input and API Search Endpoint

**Kind**: Story
**Status**: Ready
**Priority**: High
**Author**: Toby Clemson

## Summary

As a user navigating the visualisation app, I want a persistent search box in the Sidebar with a `/` keybind, so I can find documents across all twelve LIBRARY doc types without leaving my current view.

This story adds a new server endpoint `GET /api/search?q=<query>` returning matches across the twelve LIBRARY doc types, and wires the Sidebar search input slot created by 0053: a `/` keybind activator (early-returning while a text input, textarea, or contenteditable element has focus), a 200 ms debounced `useSearch` hook with 2-character minimum, inline result rendering, and a no-match empty state.

## Context

Child of 0036 â€” Sidebar Redesign. 0036 is the bundled epic; this story owns the search input behaviour and the `/api/search` backend. The Sidebar layout, search input slot, and `/` hint chip are delivered by 0053 (sibling). The activity feed lands in 0055 (sibling).

We refer to the twelve non-`Templates` variants of `DocTypeKey` collectively as the **LIBRARY doc types**. `DocTypeKey` is defined in `server/src/docs.rs:6-20` (the canonical enum used across the indexer, API, and frontend type bridge); its twelve LIBRARY variants are `Decisions`, `WorkItems`, `Plans`, `Research`, `PlanReviews`, `PrReviews`, `WorkItemReviews`, `Validations`, `Notes`, `PrDescriptions`, `DesignGaps`, and `DesignInventories` (kebab-case wire forms: `decisions`, `work-items`, `plans`, `research`, `plan-reviews`, `pr-reviews`, `work-item-reviews`, `validations`, `notes`, `pr-descriptions`, `design-gaps`, `design-inventories`). Any change to that enum's variant set must also revisit this story's endpoint contract.

There is no existing `/api/search` route in `server/src/api/mod.rs:24-46`, no client fetch helper in `api/fetch.ts`, and no query-key entry in `api/query-keys.ts`. The pre-existing `<input type="search">` in `LifecycleIndex.tsx:62-69` is a pure client-side substring filter over already-loaded data and is unrelated to the new Sidebar search feature introduced by this story.

The Claude design prototype includes a reference implementation of this feature: `useDebouncedValue` (200 ms trailing-edge), `useSearch`, `SearchBox` (sidebar input + inline results + `/` keybind), and `buildCorpus`/`rankCorpus` in `src/search.jsx`, wired into the prototype's `Sidebar`. Treat the prototype as the design and behaviour reference â€” the public shapes and observable behaviour in this story match it. See References.

## Requirements

This story owns the search backend endpoint and the Sidebar search input behaviour.

- Add a new server route `GET /api/search?q=<query>` registered in `server/src/api/mod.rs:24-46`, returning matches across the twelve LIBRARY doc types (every `DocTypeKey` variant except `Templates`). Response shape: `{ results: Array<{ docType: DocTypeKey, title: string, slug: string | null, path: string, mtimeMs: number }> }`. Only the fields `title`, `slug`, and `body_preview` are searched server-side; `path` and `rel_path` are not searched. Matching is case-insensitive throughout. Results are ordered server-side per the ranking strategy pinned in Acceptance Criteria (exact-slug â†’ title/slug prefix â†’ title/slug interior â†’ body-preview interior, each bucket by `mtime_ms` desc, ties on `path` asc). The `mtimeMs` field is returned so clients can verify ordering and so future stories can introduce client-side re-ranking without a contract change.
- Wire the Sidebar search input slot (delivered by 0053) with a global `/` keydown listener in `RootLayout.tsx` (hereafter "the keybind listener") that focuses the Sidebar search input. The keybind listener early-returns (does not steal focus or call `preventDefault`) while a text input, textarea, or contenteditable element has focus.
- Implement a `useSearch(query)` hook with 200 ms client-side debounce and 2-character minimum length. The React Query cache key is `queryKeys.search(<settled-query>)`, where `<settled-query>` is the trimmed query value present 200 ms after the most recent keystroke (not the raw input value).
- Add a new query-key `queryKeys.search(query)` entry in `frontend/src/api/query-keys.ts` and a new fetch helper `fetchSearch(query)` in `frontend/src/api/fetch.ts`.
- Render results inline beneath the search input as `<a href>` link elements (one per result), each row showing the per-doc-type Glyph, the document title, and the doc-type label. Native browser link semantics apply â€” modifier-click / middle-click open in a new tab, keyboard Enter on a focused row navigates.
- Render a no-match empty state when `results` is an empty array, as an element with `role="status"` containing the literal text `No matches` (see Acceptance Criteria for the queryable signal).
- State ownership: `frontend/src/components/Sidebar/Sidebar.tsx` owns the controlled query input state and the results panel state. `RootLayout.tsx` owns the global `keydown` listener and the `searchInputRef`, passed into `Sidebar` by prop (no new React Context, no module-level singleton).

## Acceptance Criteria

- [ ] Given the user presses `/` and no text input, textarea, or contenteditable element currently has focus, when the keydown fires, then focus moves to the Sidebar search input and the `/` character is not inserted into any field.
- [ ] Given a text input, textarea, or contenteditable element currently has focus, when the user presses `/`, then the `/` character is inserted normally into that element and Sidebar search focus does not change.
- [ ] Given no text input, textarea, or contenteditable element has focus, when the user presses `/` while holding any of Cmd, Ctrl, Alt, or Meta, then focus does not move to the Sidebar search input and `preventDefault` is not called; the keystroke proceeds with native browser semantics (so existing browser/IME shortcuts on modifier+`/` are unaffected).
- [ ] Given the user types into the search input, when the trimmed query value present 200 ms after the most recent keystroke (the "settled query") has length â‰¥ 2, then `GET /api/search?q=<settled-query>` is called exactly once per distinct settled-query value. The debounce is trailing-edge: every keystroke resets the 200 ms timer, and the request fires only once no further keystrokes have arrived for 200 ms. Concrete examples: (a) typing `ab`, deleting to `a`, then retyping `ab` while the React Query cache entry for `queryKeys.search('ab')` is still live yields exactly one network request for `q=ab` (the second occurrence is a cache hit). For this criterion, "still live" means the retype arrives within ~1 second of the first response, well inside React Query's default `gcTime` of 5 minutes. (b) typing `ab`â†’`abc`â†’`ab` with each keystroke â‰¤ 100 ms after the prior yields exactly one network request for `q=ab` (and zero requests for `q=abc`, because `abc` was never the settled value for 200 ms).
- [ ] Given `GET /api/search` responds with `{ results: Array<{ docType: DocTypeKey, title: string, slug: string | null, path: string, mtimeMs: number }> }` and the response is non-empty, when the response resolves, then after `await waitFor(() => screen.findAllByRole('link'))` the results area contains exactly `results.length` link elements whose `href` attributes equal the per-row `/library/<docType>/<slug>` values in response-array order, each rendered with the per-doc-type Glyph, the document title, and the doc-type label (the doc-type label text source is the same `docTypeMeta`-style mapping used elsewhere in the app â€” implementer's choice of mapping module, but consistent with the existing labels rendered in `LibraryTypeView`). No intermediate `No matches` element is observable between request issue and results-rendered (the in-flight transition contract â€” clearing previous results and not showing `No matches` while pending â€” is pinned in the dedicated in-flight criterion below).
- [ ] Given the response contains matches drawn from multiple ranking buckets, when results render, then rows appear in the same order as the response array, which is ordered server-side as: (1) exact-slug match (case-insensitive) â†’ (2) title-or-slug prefix match (case-insensitive) by `mtime_ms` desc â†’ (3) title-or-slug interior-substring match (case-insensitive) by `mtime_ms` desc â†’ (4) `body_preview` interior-substring match (case-insensitive) by `mtime_ms` desc. Within every bucket, ties on `mtime_ms` are broken by `path` ascending.
- [ ] Given the query `q` matches the `slug` of exactly one entry case-insensitively (e.g. typing `0054` matches a work item with slug `0054-sidebar-search`), when results are ranked, then that entry appears first in the response array regardless of its `mtime_ms` relative to other matches.
- [ ] Given an entry has `q` as a case-insensitive prefix of its `title` or `slug`, when results are ranked, then that entry is ordered before any entry where `q` appears only as a case-insensitive interior substring of `title` or `slug` (regardless of relative `mtime_ms`).
- [ ] Given an entry has `q` matching only as a case-insensitive substring of its `body_preview` (and not in `title` or `slug`), when results are ranked, then that entry is ordered after every entry matching in `title` or `slug` regardless of `mtime_ms`.
- [ ] Given an entry whose `title`, `slug`, and `body_preview` are all case-insensitive substring-free of `q`, when `/api/search?q=<q>` is called, then that entry is absent from `results`. (The set of searched fields is pinned in Requirements.)
- [ ] Given a rendered result row, when the user clicks it (plain click), then the browser navigates to the row's library doc detail route. The row is rendered as an `<a href="/library/<docType>/<slug>">` element where `<docType>` is the kebab-case wire form of the `DocTypeKey` (e.g. `work-items`, `decisions`) and `<slug>` is the row's `slug` â€” matching the existing `libraryDocRoute` shape `/library/$type/$fileSlug` in `frontend/src/router.ts`. Native link semantics apply: modifier-click and middle-click open a new tab, and pressing Enter while the row is keyboard-focused triggers the same navigation as a plain click. Rows where `slug` is `null` are not rendered as result links (such entries cannot satisfy the slug-based ranking buckets either).
- [ ] Given the user types fewer than 2 characters into the search input, when 200 ms elapses, then no `/api/search` request is issued and no results are rendered.
- [ ] Given previously rendered results are visible (a prior settled query of length â‰¥ 2 succeeded), when the user edits the query so that the settled query falls below 2 characters, then the results area is cleared â€” no result rows remain and the `No matches` element is not shown.
- [ ] Given the user types a query that returns no matches, when `/api/search` responds with `results: []`, then the results area shows an empty-state element with `role="status"` containing the literal text `No matches`, queryable via `getByRole('status')`.
- [ ] Given a settled query of length â‰¥ 2 has triggered a request that has not yet resolved, when the results area renders, then it does not show the `No matches` element and does not show rows from an arbitrarily-older settled query â€” only one of the following transitional contents is permitted: (a) an empty results area, or (b) the rows from the *immediately prior* settled query held as React Query placeholder data (`placeholderData: keepPreviousData`, distinguishable via `search.isPlaceholderData === true`). Once the new settled query resolves, the placeholder rows are replaced by the new results (or by the `No matches` element if the new result set is empty). This admits the no-flicker UX shape where the prior key's rows stay visible until the next response arrives, while still forbidding the AC's load-bearing prohibitions: no `No matches` flash mid-transition, no rows that belong to neither the current nor the immediately prior key.
- [ ] Given `GET /api/search` responds with a non-2xx status or the network request fails, when the failure is observed, then the results area is cleared (no stale rows, no `No matches` element) and the failure is logged via `console.error` with a `FetchError` instance whose message includes the substring `/api/search` (consistent with the existing `FetchError` handling in `frontend/src/api/fetch.ts`). Verifiable as `vi.spyOn(console, 'error')` having been called with a first argument `err` satisfying `err instanceof FetchError && err.message.includes('/api/search')`. No user-visible error banner is required by this story.
- [ ] Given an indexer snapshot (the `Vec<IndexEntry>` returned by `Indexer::all()`) containing entries across LIBRARY doc types whose `title` or `slug` substring-matches `q`, when `GET /api/search?q=<title-substring>` is called server-side, then the response is `{ results: [...] }` containing matches from the LIBRARY types and zero entries with `docType === "templates"`. (The handler relies on `Indexer::all()` already excluding Templates by structural omission â€” see Technical Notes; this criterion guards against a future indexer change that starts surfacing Templates entries by also asserting the absence of `docType === "templates"` in the response regardless of `Indexer::all()`'s contents.)
- [ ] Given `q` is absent or an empty string, when `GET /api/search` is called, then the response is `{ results: [] }` with HTTP 200 (no error, no 4xx).

## Assumptions

- The `/api/search` endpoint is implemented within this story (not coordinated as parallel backend work).
- A client-side debounce of 200 ms with a 2-character minimum is sufficient to keep request rate acceptable without a dedicated rate limiter on the server.
- The four-bucket case-insensitive ranking strategy (exact-slug â†’ title/slug prefix â†’ title/slug interior â†’ body-preview interior) is a reasonable first cut for production. The Claude design prototype's `rankCorpus` uses an equivalent strategy and it has read well in the prototype; production usage may reveal a need to revisit (e.g. add fuzzy matching, weight recency more aggressively, or include `path` as a searched field). No follow-up story is currently tracked; if quality issues surface in production, capture a new work item at that point rather than pre-creating a placeholder.

## Technical Notes

**Size**: M â€” server route + handler (small, well-precedented by api/docs.rs), useSearch + fetchSearch + queryKeys.search (small, well-precedented), useDebouncedValue helper (net-new but ~10 lines), global `/` keybind via RootLayout's first useEffect (net-new infra, small), Sidebar gains its first local state + results panel (small-medium). Each piece is contained; the M rating reflects that two pieces of frontend infrastructure are net-new (debounce, hotkey) rather than any single piece being large.

- **Prototype reference**: the Claude design prototype implements this feature end-to-end. The inventory at `meta/research/design-inventories/2026-05-21-015231-claude-design-prototype/inventory.md:392-400` describes the relevant symbols: `useDebouncedValue` (200 ms trailing-edge), `useSearch` (97 lines, returns `{ status, results }`), `SearchBox` (137 lines â€” sidebar input + inline results panel + `/` global keybind), and `buildCorpus`/`rankCorpus` (19 / 52 lines). The public shapes in this story (`useSearch(query)`, debounce delay, keybind behaviour) match the prototype. **Note**: `src/search.jsx` is referenced by the inventory but does **not** exist on disk; the standalone prototype HTML at `prototype-standalone.html` does not embed the search source either. Implementers must work from the inventory's symbol descriptions plus the rendered prototype, not a readable JSX source. Where this story's Acceptance Criteria and Requirements differ from observed prototype behaviour, the ACs and Requirements are authoritative â€” the prototype is a design reference, not a behavioural spec.
- **Frontend wiring**:
  - New `useSearch(query: string)` hook: internally pipes the raw `query` argument through `useDebouncedValue(query.trim(), 200)`, then uses `useQuery` with `queryKey: queryKeys.search(debouncedQuery)` and `queryFn: () => fetchSearch(debouncedQuery)`, `enabled: debouncedQuery.length >= 2`. The cache key is always the settled (trimmed, debounced) value â€” not the raw input â€” so the dedup behaviour in the debounce settled-query criterion follows directly from React Query's default key-based caching. Use the global `QueryClient` defaults (no per-query `staleTime` or `gcTime` overrides); the debounce settled-query criterion's "still live" window means "within the React Query defaults" â€” i.e. the cache entry has not been garbage-collected (`gcTime` default 5 minutes) since its last subscriber unmounted. Verification tests should re-trigger the debounce settled-query criterion example within ~1 second of the initial response so the cache entry is unambiguously live. Add `search: (q: string) => ['search', q] as const` to `queryKeys` in `skills/visualisation/visualise/frontend/src/api/query-keys.ts:40-62` and `fetchSearch(q: string): Promise<SearchResult[]>` (where `SearchResult` matches the response row shape including `mtimeMs`) in `frontend/src/api/fetch.ts` following the `fetchTypes` exemplar (lines 59-64): bare `fetch()`, throw `FetchError` on `!r.ok`, parse inline-typed body, return inner field.
  - **Debounce: net-new infrastructure.** No `useDebounce` / `useDebouncedValue` exists anywhere in `frontend/src`. Introduce a named `useDebouncedValue(value, delayMs)` helper (â‰ˆ10 lines using `useState` + `useEffect` + `setTimeout`/`clearTimeout`) co-located with `useSearch`. Keep it as a named exported helper rather than inlined â€” this single shape pre-empts confusion in review.
  - **Infrastructure follow-up posture.** Both `useDebouncedValue` and the global `/` keybind are introduced as small, single-consumer primitives in service of this feature (not as a generic platform layer). If a second consumer for either appears during implementation, split the generalisation out as a follow-up chore rather than expanding scope inside this story.
  - **`/` keybind listener: net-new infrastructure.** No `useHotkey`, `useKeybind`, or global `keydown` listener exists anywhere in `frontend/src`. Add the keybind listener as a global `keydown` `useEffect` in `RootLayout.tsx` â€” this will be the **first `useEffect` in that file** (RootLayout currently has zero effects and zero event listeners). The keybind listener checks `document.activeElement` against `{ INPUT, TEXTAREA }` tags and `isContentEditable` before stealing focus; on `/` (and no modifier keys â€” Cmd, Ctrl, Alt, or Meta held suppresses activation and lets the default browser behaviour proceed), `preventDefault()` and focus the Sidebar search input via the `searchInputRef` prop passed from `RootLayout` into `<Sidebar>`. **Plumbing decision**: `RootLayout` owns the `searchInputRef` and the keybind listener, and passes the ref into `<Sidebar>` by prop. No new context, module-level singleton, or other shared object is introduced (deferred until a second consumer exists).
- **Sidebar input host**: `frontend/src/components/Sidebar/Sidebar.tsx` is purely presentational today with no local state. The uncontrolled search input slot from 0053 is already rendered at `frontend/src/Sidebar/Sidebar.tsx:22-33` with a `TEMPORARY` comment naming this work item. This story introduces the first state hook in this component â€” `Sidebar` owns the controlled query string and the rendered results panel state; `RootLayout` retains ownership of the focus ref and global listener (see the `/`-keybind listener plumbing decision earlier in Frontend wiring). The only new prop from `RootLayout` is `searchInputRef: RefObject<HTMLInputElement>`; the controlled `value`/`onChange` stay internal to `Sidebar`. The `LifecycleIndex.tsx:62-69` precedent uses `<input type="search">` with `aria-label` and `placeholder` but is purely synchronous (no debounce, no 2-char minimum) â€” reuse the markup conventions, not the filter loop.
- **`/` hint chip**: the chip rendered in the search input slot is delivered and owned by 0053. This story does not modify the chip's appearance, visibility, or dynamic behaviour â€” the chip is purely informational visual state owned by 0053.
- **Server route**:
  - Register `GET /api/search` in `skills/visualisation/visualise/server/src/api/mod.rs:24-46` â€” add `mod search;` to the module list (alphabetically between `related` and `templates`) and `.route("/api/search", get(search::search))` to the existing chain. The Router is `Router<Arc<AppState>>` (`mount()` at line 24).
  - **Handler precedent**: model on `api/docs.rs:30-41` â€” `pub(crate) async fn search(State(state): State<Arc<AppState>>, Query(q): Query<SearchQuery>) -> Result<Response, ApiError>` â€” `docs_list` is the closest precedent because it combines `State` + `Query` and returns `Result<Response, ApiError>`. `SearchQuery` is a private struct with `q: String`.
  - **DocTypeKey wire format**: `DocTypeKey` carries `#[serde(rename_all = "kebab-case")]` (`server/src/docs.rs:5`), so e.g. `WorkItems` serialises as `"work-items"`. Response rows should use camelCase field names (`docType`, `mtimeMs`) per the existing `DocType` precedent (`docs.rs:134-157`) â€” i.e. the row struct is `#[serde(rename_all = "camelCase")]` and its `r#type: DocTypeKey` field is renamed `docType`. The `DocTypeKey` token in the AC examples (`"templates"`, `"work-items"`) is the kebab-case wire form, not the Rust variant name.
  - **Index access**: read `state.indexer.all()` (`indexer.rs:617-619`) â€” the natural entry point for global search. It's `async fn` returning `Vec<IndexEntry>` (clones every value under a read lock), so the handler must `.await` it and pays O(N) per request over a freshly cloned snapshot â€” acceptable given the 200 ms client-side debounce and 2-char minimum. There is **no `entries_by_type` map** on the Indexer; the primary store is `entries: Arc<RwLock<HashMap<PathBuf, IndexEntry>>>` keyed by path (`indexer.rs:216`), and `all_by_type` (`indexer.rs:532-540`) is itself a linear filter. `IndexEntry` (`indexer.rs:162-192`) carries `title: String`, `slug: Option<String>`, `path: PathBuf`, `rel_path: PathBuf`, `mtime_ms: i64`, and `body_preview: String` â€” all three searched fields are reachable without extending the entry shape. Note `slug` is optional: use `entry.slug.as_deref().unwrap_or("")` when matching, and treat absent slugs as non-matches for the exact-slug and slug-prefix/interior buckets.
  - **Templates exclusion**: Templates are explicitly skipped during the indexer's per-kind enumeration (`indexer.rs:293-300`) and never enter `entries`; they're managed by `server/src/templates.rs` separately. `Indexer::all()` therefore returns zero Templates entries by structural omission (asserted by `indexer.rs:1444`), so the handler needs no `kind != Templates` predicate.
  - **Ranking strategy** (pinned per Acceptance Criteria): four-bucket case-insensitive ordering â€” (1) exact-slug match â†’ (2) title-or-slug prefix â†’ (3) title-or-slug interior substring â†’ (4) `body_preview` interior substring. Within each bucket, order by `mtime_ms` descending; ties broken by `path` ascending. This matches the prototype's `rankCorpus` behaviour (see Prototype reference).
- **Search input slot**: delivered by 0053 â€” `<input type="search">` slot with `/` hint chip already in DOM at `frontend/src/Sidebar/Sidebar.tsx:22-33`. This story wires `value`/`onChange`, attaches the `searchInputRef` prop (passed from `RootLayout`) to the input element, the global `/` listener (in `RootLayout`), and the results panel beneath it.
- **Glyph for result rows**: `Glyph` (`Glyph/Glyph.tsx:39-49`) accepts `{ docType: DocTypeKey, size: 16 | 24 | 32, ariaLabel?, framed? }` directly. Result rows use `<Glyph docType={result.docType} size={16} framed />` â€” framed (per the per-doc-type framing convention used by `EyebrowLabel.tsx:13` and Library/Overview cards), decorative, no `ariaLabel`; the row's own link text and doc-type label carry the accessible name.
- **No `api/activity.rs`**: if you look for sibling handler precedents, `api/lifecycle.rs` and `api/docs.rs` are the right examples â€” there is no `server/src/api/activity.rs` (activity is a non-HTTP module at `server/src/activity.rs`).
