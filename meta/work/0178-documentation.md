# 0178: Documentation

**Kind**: Epic
**Status**: Draft
**Priority**: Medium
**Author**: Phil Helm

## Summary

As a reader of Accelerator's documentation, I want docs that serve my
current mode — evaluating, starting, doing, looking up, or contributing —
so that I can succeed with the plugin without the authors present.

Umbrella epic for all documentation work: the Astro Starlight docs site,
the README, and the in-repo `docs/` tree. Individual documentation
efforts land as child work items under this epic.

## Context

The docs site (Astro Starlight, published to GitHub Pages) is standing
(0177) after the README was slimmed and split into a `docs/` tree (0175)
and per-skill-family reference docs were authored (0176). This epic
groups that work and everything that follows under one umbrella with
shared goals and audiences.

Documentation must serve five audiences:

1. **Evaluators/deciders** — deciding whether to adopt Accelerator; may
   never install it. Need the pitch, philosophy, demos, and confidence
   signals (releases, compatibility, maintenance).
2. **Newcomers** — have decided to try it; need install and a guided
   first loop.
3. **Practitioners** — have it and want to accomplish tasks and learn to
   use it really well; need task-oriented guides and deeper explanation
   of the design.
4. **Deep-divers** — need to look up any of the ~82 skills, agents, and
   internals in a complete, uniform reference.
5. **Contributors** — want to hack on Accelerator itself; need
   contributor and internals documentation.

The Diátaxis framework (tutorials / how-to / reference / explanation) is
an organising influence for mapping these audiences to page types, not a
mandate.

## Requirements

High-level goals:

- Each audience has a clear entry point on the docs site and can achieve
  their goal using only the docs.
- Documentation lives on the Starlight site; the README stays slim and
  links in.
- Docs stay accurate as the plugin evolves — prefer generation and drift
  guards over hand-maintained duplication wherever feasible.

Initial children:

- 0177 — Stand up a documentation site for the docs/ tree (draft)
- 0145 — Documentation Improvements (in-progress)
- (new) Make the docs amazing — content, structure, generated skill
  reference, and polish overhaul of the Starlight site

## Acceptance Criteria

- [ ] Given an evaluator who has never installed Accelerator, when they
      read the docs site, then they can understand what it is, why it is
      designed this way, and see it demonstrated well enough to make an
      adoption decision.
- [ ] Given a newcomer, when they follow the docs site alone, then they
      can install the plugin and complete a first research → plan →
      implement loop.
- [ ] Given a practitioner with a task in mind, when they consult the
      docs, then a task-oriented guide or skill-discovery path gets them
      to the right skill; and material exists that deepens their command
      of the workflow beyond the basics.
- [ ] Given any skill shipped in `plugin.json`, when a deep-diver looks
      it up, then a complete reference page exists for it.
- [ ] Given a would-be contributor, when they read the contributor docs,
      then they can set up the toolchain and land a change.
- [ ] All documentation work items are parented under this epic.
- [ ] Docs deploys and `mise run check` remain green as children land.

## Technical Notes

- Starlight supports mixing a manual sidebar with autogenerated,
  collapsed reference groups; per-page frontmatter controls sidebar
  label/order/badges — suited to generated per-skill pages.
- Ecosystem-standard approach for generated reference pages is a
  prebuild script writing markdown into the content collection (cf.
  starlight-typedoc); fits the tasks/ toolchain.
- GitHub Pages gotcha: markdown and hero.actions links are not
  base-prefixed automatically (base: /accelerator);
  starlight-links-validator is already wired in.
