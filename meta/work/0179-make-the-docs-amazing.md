# 0179: Make the Docs Amazing

**Kind**: Story
**Status**: Draft
**Priority**: Medium
**Author**: Phil Helm

## Summary

As a reader of the Accelerator docs site, I want the content, structure,
reference coverage, and presentation overhauled, so that whichever
audience I belong to (per epic 0178), the site gets me to my goal
without the authors present.

Overhaul of the Astro Starlight docs site: new tutorial and how-to
content, generated per-skill reference pages, richer explanation pages,
restructured navigation, and Starlight polish.

## Context

The docs site is standing (0177): 17 pages (~1,500 lines) on Starlight,
deployed to GitHub Pages under base `/accelerator`, with mermaid support
and starlight-links-validator wired in. Gaps: no Getting Started page
(install instructions live only in the README), 69 skills covered by
eight family pages with no per-skill reference, a thin Philosophy page,
a default (non-splash) landing page, a flat sidebar, and only one
diagram.

Research grounding (see References): the Diátaxis framework maps the
epic's audiences to page types (tutorials / how-to / reference /
explanation); Starlight supports mixing a manual sidebar with
autogenerated collapsed reference groups controlled by per-page
frontmatter; the ecosystem-standard way to generate reference pages
from source-of-truth files is a prebuild script writing markdown into
the content collection.

## Requirements

Organised by the audiences defined in epic 0178:

**Newcomers (tutorial)**

- Getting Started page: prerequisites (including minimum Claude Code
  v2.1.144), marketplace install, `/accelerator:init`, and a guided
  first research → plan → implement loop showing what the user actually
  sees at each step.

**Practitioners (how-to)**

- Task-oriented how-to guides (e.g. plan a feature, review a PR, capture
  a decision, sync work items with Linear/Jira).
- A "which skill do I need?" decision index mapping common intents to
  skills.
- Configuration cookbook: complete `.accelerator/config.md` recipes for
  common customisations.
- FAQ/troubleshooting: binary download/checksum failures, the bash 3.2
  floor, migration prompts, configuration not being picked up.

**Deep-divers (reference)**

- One generated docs page per skill (69) built from each `SKILL.md`'s
  frontmatter and body by a prebuild script in the `tasks/` invoke
  toolchain; grouped by category in a collapsed autogenerated sidebar
  group; existing family pages become curated overviews linking in.
- Agents reference page: each agent's role and the locator/analyser
  split.
- `meta/` directory anatomy page: what lives where, which skills
  read/write each path, lifecycle.

**Evaluators and power users (explanation)**

- Flesh out Philosophy into a full treatment of the problem (context
  rot) and the design response (filesystem phases, bounded contexts).
- Worked end-to-end case study: a real small feature through research →
  plan → implement → review → commit → PR with `meta/` excerpts and
  visualiser screenshots.
- Splash landing page: hero (tagline, actions), CardGrid of key
  features, LinkCards into each section.
- Expanded visualiser page: screenshots of each view (light and dark),
  ideally a GIF/video of live reload.

**Structure and polish**

- Sidebar restructured into Start Here / Guides / Reference groups.
- Asides (note/tip/caution) for gotchas and `<Steps>` for sequences.
- `lastUpdated: true` (CI checkout needs `fetch-depth: 0`), `editLink`,
  social links, favicon and og metadata.
- Mermaid diagrams: full workflow flowchart, skill-invocation sequence
  (`!` preprocessor → context injection → agent fan-out), work-item
  state machine.

**Drift guards**

- CI check that every skill registered in `.claude-plugin/plugin.json`
  has a docs page.
- README getting-started made link-only or verified against the docs
  version.

## Acceptance Criteria

- [ ] Given a newcomer with Claude Code installed, when they follow only
      the Getting Started page, then they can install the plugin and
      complete a first research → plan → implement loop.
- [ ] Given any skill registered in `.claude-plugin/plugin.json`, when
      the site builds, then a reference page generated from its
      `SKILL.md` exists, and CI fails if any skill lacks one.
- [ ] Given an evaluator on the landing page, when it renders, then it
      uses the splash template with hero actions and links into every
      sidebar section.
- [ ] Given a practitioner with a common task, when they consult the
      decision index, then it names the skill to invoke for that task.
- [ ] Given the built site, then starlight-links-validator passes, the
      GitHub Pages deploy succeeds, and `mise run check` is green.

## Assumptions

- Generated pages replace hand-written per-skill docs entirely; the
  eight family pages are retained as curated overviews, not deleted,
  and the hand-written "All Skills" index is replaced by a generated
  index.
- The generation script lives in the `tasks/` invoke toolchain and runs
  as a prebuild step, not as an Astro content loader.
- Generated pages are **build-time only** (gitignored output inside the
  content collection), not committed — drift is impossible by
  construction and the CI guard reduces to a coverage check in the
  generator.
- The 23 `user-invocable: false` skills (18 review lenses, 3 review
  output-formats, browser-executor, paths) get pages in a separate
  collapsed "Internal" sidebar group with an Internal badge.
- Frontmatter parsing uses PyYAML (`safe_load`), version-pinned in
  `pyproject.toml` per repo convention.
- The generator excludes stray `SKILL.md` files under
  `skills/visualisation/visualise/frontend/node_modules/**` and
  `skills/config/migrate/scripts/test-fixtures/**`; the real count is
  69 skills, and pages are generated as `.md` (not `.mdx`) because
  skill bodies contain MDX-hostile placeholders.

## Technical Notes

- Prebuild generation is preferred over Astro custom content loaders:
  Starlight's remark/rehype pipeline only runs on content loaded via its
  own `docsLoader()`, so loader-rendered markdown skips its transforms.
- Per-page sidebar frontmatter (`sidebar.label`, `sidebar.order`,
  `sidebar.badge`) keeps the sidebar config tiny for 69 generated pages.
- GitHub Pages gotcha: markdown links and `hero.actions` links are not
  automatically prefixed with `base: /accelerator` — prefix manually.
- `lastUpdated` reads git history: the Pages workflow checkout needs
  `fetch-depth: 0`.
