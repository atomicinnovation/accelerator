# Repo-Wide Linting, Formatting, And Static Analysis Guardrails Implementation Plan

## Overview

Establish durable lint / format / type-check guardrails across the whole
repository — the visualiser frontend (TypeScript/React via Biome + tsc), the
visualiser server (Rust via rustfmt + clippy), the Python invoke tooling (ruff
+ pyrefly), and shell scripts (shellcheck + shfmt) — and fix every existing
violation so each tool passes cleanly from the moment its check becomes
blocking. Tooling is wired into a structured `mise` task tree and into Main CI
as separate, blocking, per-component jobs.

## Current State Analysis

The repo is polyglot and only partially guarded today:

- **Frontend** (`skills/visualisation/visualise/frontend/`): no Biome / ESLint
  / Prettier / stylelint. The only gates are `tsc -b` inside `npm run build`
  and Vitest. The `typecheck` script (`package.json:12`) is a **no-op** — it
  runs `tsc --noEmit` against the solution `tsconfig.json` whose `files: []`
  checks nothing. `tsconfig.e2e.json:7,13` has a **duplicate `"types"` key**.
  Prevailing style is single-quote / no-semicolon / 2-space.
- **Server** (`skills/visualisation/visualise/server/`): no `[lints]` tables,
  no `rustfmt.toml`/`clippy.toml`, no fmt/clippy task or CI step. `Cargo.toml`
  is regenerated by `tasks/version.py` on version bumps (verified to use a
  `tomlkit` round-trip — see Key Discoveries). Feature flags fragment clippy
  coverage; `build.rs` panics without `frontend/dist/` when `embed-dist` is on.
- **Python** (repo root): **63 tracked `.py` files** (not "~700+" — that figure
  counted `.venv/` site-packages). `pyproject.toml` has pytest/uv config but no
  ruff/pyrefly/mypy. Code is consistently typed with modern 3.14 syntax; 7
  `# noqa` markers exist (one already using a ruff code).
- **Shell** (173 tracked `.sh` + 1 extensionless script): **already half-built**
  — shfmt (`format:scripts:check`), shellcheck (`lint:scripts:shellcheck:check`,
  `-x --severity=warning`), and a bashisms guard (`lint:scripts:bashisms:check`)
  already run under `mise run check` in CI. Scope is centralised in
  `tasks/shared/sources.py` (161 files; excludes fixtures + `test-helpers.sh`).
- **mise/CI**: established conventions — colon-separated task names ending
  `:check`/`:fix`, `description` on every task, aggregates as pure `depends`
  fan-outs, implementations delegated to Python invoke; every CI job is
  `checkout → jdx/mise-action → mise run <aggregate>`.

## Desired End State

A consistent, three-family `mise` task tree (`format:*`, `lint:*`, `types:*`)
with per-component and top-level aggregates; committed tool configs at the
mandated strictness; every existing violation fixed; every suppression and
exclude carrying an adjacent justification comment; and Main CI running each
component's checks as a separate blocking job. Verifiable by: `mise run check`
exits zero locally; each per-component CI job exits non-zero when fed a
representative violation (falsification probe); and tool file-sets cover every
tracked `.py`/`.sh` outside `workspaces/`.

### Key Discoveries

- **Cargo.toml templating is safe**: `tasks/version.py:31-34` `_render_cargo_toml`
  uses `tomlkit.parse` → `tomlkit.dumps`, a round-trip that **preserves unknown
  tables and comments**. A `[lints.clippy]` table survives version bumps with no
  `version.py` change (locked in by a new guard test in Phase 4).
- **tsc is a no-op today**: `tsconfig.json:3-7` is a solution file (`files: []`);
  the standalone type-check must be `tsc -b --noEmit`. `tsconfig.app.json:13`
  already sets `noEmit: true` and `build` already runs `tsc -b`, so `-b --noEmit`
  is viable on the pinned TS (≥5.6).
- **Biome 2.4 specifics** (researched): `--error-on-warnings` is **CLI-only**
  (no config key) — it lives in the committed mise task. Stock defaults give
  double-quote + semicolons + `trailingCommas: "all"`, but default `indentStyle`
  is **tab** and `.editorconfig` is read only when `formatter.useEditorconfig:
  true`. `files.includes` is the v2 single-array form with `!`-negation. CSS
  *formatter* is **off by default** (`css.formatter.enabled` must be set). Config
  needs comments → use **`biome.jsonc`**.
- **Clippy**: `-D warnings` **cannot** be expressed in `[lints.clippy]`; it stays
  on the CLI (consistent with the work item's "CI enforces with `-D warnings`").
  `pedantic = { level = "warn", priority = -1 }` lets curated `allow`s
  (priority 0) override it.
- **`--all-features` hides code**: it enables `dev-frontend`, compiling out the
  `cfg(not(feature = "dev-frontend"))` arms in `src/assets.rs`. Full coverage
  needs a second default-feature clippy pass (mirrors the two-pass unit tests at
  `tasks/test/unit.py:6-24`). Both passes enable `embed-dist`, so both need
  `frontend/dist/index.html` to *exist* — but `build.rs` only checks existence
  (it does not read contents), so a lightweight stub suffices for a lint compile
  rather than a full `build:frontend` (see Phase 4, `build:frontend:stub`).
- **ruff/rustfmt ignore `.editorconfig`** — line-length is duplicated in their
  configs with a cross-reference comment. Biome and shfmt read it natively.
- **pyrefly install path is the one unknown**: confirm `mise registry | grep
  pyrefly` at implementation; fall back to a `pipi:pyrefly`/`cargo:pyrefly`
  backend pinned exact.

## What We're NOT Doing

- **Not** adopting the work item's literal flat task names (`lint:frontend`,
  `lint:rust`, …). Per the author's structure decision, type-checks get their
  own `types:*` family and the tree mirrors the existing `format:*`/`lint:*`
  split. The AC→task mapping is documented in Phase 6 so validation stays
  traceable.
- **Not** renaming the existing `server`/`scripts` component labels to
  `rust`/`shell` (avoids churning established `build:server:*` /
  `format:scripts:*` tasks).
- **Not** adding pre-commit hooks (CI-only enforcement, per the work item).
- **Not** covering Markdown / YAML / TOML / JSON-outside-frontend / GitHub
  Actions YAML, SCSS, or stylelint (explicitly out of scope).
- **Not** adding cargo/npm/uv build caching to CI (optional; out of scope).
- **Not** consolidating 0090's radius-token Vitest gate into the lint structure
  (left to future discretion per the work item).
- **Not** touching the convention-enforcement test gates (design-token /
  template / frontmatter suites) — they remain authoritative alongside the new
  tooling.

## Implementation Approach

Six phases. Phase 1 lays shared foundations. Phases 2–5 are one self-contained,
independently-mergeable per-component slice each (config + task wiring + full
remediation sweep + blocking CI job + falsification probe). Phase 6 wires the
top-level aggregates, documents the AC mapping, and raises the pyrefly follow-up.

Recommended landing order: 1 → 2 (shell, closest to done) → 3 (build-system, small) →
4 (server) → 5 (frontend, largest sweep) → 6. Phases 2–5 are mutually
independent and may land in any order; each depends only on Phase 1's
`.editorconfig` anchor (a soft dependency — tools hard-code the matching value,
so order only affects whether the cross-reference comment's target exists yet).

**Per-component rhythm (TDD-shaped):**
1. Add the tool config + mise tasks (no CI yet).
2. **Report-only run** — run the tool in report mode; record the violation
   count and enumerate the suppressions/excludes the sweep will add. This is the
   work item's sizing checkpoint and the "red" state.
3. **Remediate to green** — mechanical auto-fixes as a distinct commit, then
   manual fixes, then justified suppressions. Audit pre-existing suppressions
   for adjacent rationale. When the sweep reformats **test** files, review the
   test-file diff specifically: a formatter rewriting an asserted command/string
   literal (e.g. quote style on an asserted `shfmt -d ...` command) can change
   what a test asserts while keeping it green, so confirm the mechanical commit
   makes no semantic change to any assertion.
4. Add the blocking CI job (`mise run <component>:check`) **and wire it into the
   `prerelease` job's `needs:` list in the same PR**. The `needs:` edit is *not*
   deferred to Phase 6: until a job is in `needs:`, it runs but does not gate the
   push-triggered `prerelease`/`release` pipeline, so "blocking the moment it
   lands" would not hold during the staged rollout. Phase 6 only *asserts* the
   list is complete. (Phase 2 already does this for `check-scripts`.)
5. **Falsification probe** — introduce one representative violation per check
   command, observe the job go non-zero, link the run from the PR, revert. While
   the probe is failing, also confirm the PR is **blocked from merge** (not just
   that the job is red) — the same verify-it-blocks discipline Phase 2 applies to
   the renamed required check, so each newly-required `check-<component>` is
   proven to actually gate. A required-check *name* added to branch protection
   that does not exactly match the emitted job name fails open silently.

Each component's `<component>:check` aggregate is added in its own phase; the
top-level `format:check`/`lint:check`/`types:check`/`check` rewire is deferred to
Phase 6 so per-component phases never touch shared aggregate lines (no merge
conflicts, any-order landing).

**Component → tools → families:**

| Component | format | lint | types |
|---|---|---|---|
| `frontend` | Biome (formatter) | Biome (linter) | tsc |
| `server` | rustfmt | clippy | — |
| `build-system` | ruff format | ruff check | pyrefly |
| `scripts` | shfmt | shellcheck + bashisms | — |

### Overview

Shared, low-risk anchors used by every later phase: the `.editorconfig`
line-width + per-language indent that ruff/rustfmt cross-reference and Biome/shfmt
read; and exact-version pins for `jdx/mise-action` and `node`. No tool behaviour
changes — independently mergeable, CI stays green.

### Changes Required

#### 1. `.editorconfig` — canonical width + per-language indent

**File**: `.editorconfig`
**Changes**: Add a `[*]` width anchor and per-language indent sections (ruff and
rustfmt duplicate the width by hand; Biome reads it via `useEditorconfig`; shfmt
already reads `[*.sh]`). A blanket `[*] indent_size` is deliberately avoided —
Python/Rust use 4-space, frontend uses 2-space.

```ini
root = true

# Canonical line width for the repo. ruff (line-length) and rustfmt (max_width)
# duplicate this value in their own config because they do not read
# .editorconfig; Biome (useEditorconfig) and shfmt read it natively. Keep the
# four in sync by hand — there is no automated consistency check (by design).
# Note: Biome also applies this width to JSON/JSONC (package.json,
# tsconfig*.json), so the one-time 80-col rewrap of those files is an accepted
# part of the frontend reformat sweep (recorded so the diff size is expected,
# not a surprise). If undesirable, override json.formatter.lineWidth in
# biome.jsonc instead — left at the canonical width by choice.
[*]
max_line_length = 80

[*.tsv]
indent_style = tab
trim_trailing_whitespace = false

# Frontend (Biome reads these via formatter.useEditorconfig). `mts`/`cts` are
# included so a future ECMAScript-module-typed file is not formatted with
# Biome's tab default (the [*] section sets no indent_style).
[*.{ts,tsx,mts,cts,js,jsx,mjs,cjs,css,json,jsonc}]
indent_style = space
indent_size = 2

# Python (informational for editors; ruff does not read .editorconfig — its
# line-length/indent are duplicated in pyproject.toml with a cross-reference).
[*.py]
indent_style = space
indent_size = 4

# Rust (informational for editors; rustfmt does not read .editorconfig — its
# max_width is duplicated in rustfmt.toml with a cross-reference).
[*.rs]
indent_style = space
indent_size = 4

# shfmt reads these when invoked with no formatting flags (single source of
# truth shared by editors and CI). switch_case_indent is the .editorconfig
# equivalent of shfmt's -ci.
[*.sh]
indent_style = space
indent_size = 2
switch_case_indent = true
```

#### 2. `tasks/lint/scripts.py` — drop CLI flags now owned by the rc

**File**: `tasks/lint/scripts.py`
**Changes**: `shellcheck -x --severity=warning {args}` → `shellcheck {args}`
(the rc supplies `external-sources` and `enable=all`; dropping `--severity`
surfaces info/style per the strict mandate). Update the docstring/`description`.
`bashisms` task unchanged. **Update the existing unit test**:
`tests/unit/tasks/test_lint.py::TestShellcheckTask.test_command` currently
asserts `cmd.startswith("shellcheck -x --severity=warning ")` — in the same
commit, change it to assert the bare `shellcheck ` prefix **plus explicit absence
checks** (`assert "-x" not in cmd` and `assert "--severity" not in cmd`) — a
`startswith`-only rewrite would still pass if a stray flag survived later in the
string. This is the unit-level proof that flag ownership moved to the rc rather
than silently vanishing. Update the `lint:scripts:shellcheck:check`
task `description` (it names `-x, --severity=warning` today) to point at
`.shellcheckrc`.

```python
@task
def shellcheck(context: Context):
    """Lint every shell source with ShellCheck (config in .shellcheckrc)."""
    args = _sources_args()
    if args is None:
        # Fail loudly, not green: an empty match set means scope discovery
        # broke (a glob/`_keep` regression), not that there is nothing to lint.
        raise Exit("shellcheck: no shell sources matched — scope discovery is broken", code=1)
    with context.cd(str(repo_root())):
        result = context.run(f"shellcheck {args}", warn=True, pty=False)
    if result.exited != 0:
        raise Exit(
            "shellcheck reported findings — fix them, or add a justified "
            "`# shellcheck disable=`/`source=` directive",
            code=1,
        )
```

The empty-set-fails-loudly guard applies to **all** shell tasks reading
`_sources_args()` (shfmt and bashisms too): replace any silent `return` on an
empty match set with a raised `Exit`, so an accidental scope collapse fails the
gate rather than passing it green (fail-closed, not fail-open). **This change
also breaks an existing test**: `tests/unit/tasks/test_format.py::TestFormatCheck::test_noop_when_no_sources`
currently asserts the *old* silent-noop behaviour (`fmt.check` does not raise and
`ctx.run` is not called on an empty set). Invert it in the same commit to assert
`fmt.check` now raises `Exit` on an empty source set — giving the fail-closed
contract a real regression test rather than a contradicting one. `test_format.py`
is the only shell task-test asserting the old empty-set behaviour, so also **add**
an empty-source-set `pytest.raises(Exit)` test to `TestShellcheckTask` and
`TestBashismsTask` (which currently only mock non-empty source lists), so the
fail-closed contract is regression-guarded across all three shell tasks rather
than just shfmt.

#### 3. `tasks/shared/sources.py` — widen scope, add the CLI script

**File**: `tasks/shared/sources.py`
**⚠ Preserve the existing `os.walk`/`pathspec` discovery — do NOT rewrite to
`git ls-files`.** The current `shell_sources()` is a deliberate gitignore-honouring
filesystem walk; its docstring records that `git ls-files` was abandoned because
it is **blind inside a jj workspace** (git resolves to the parent repo whose index
does not track the workspace), which silently emptied the scan and let unformatted
scripts reach CI. (The research doc's `git ls-files` description, and earlier
drafts of this plan, are stale — the implementation has since moved to the walk.)
A `git ls-files` rewrite would reintroduce that fail-open bug and break the five
existing walk-behaviour tests (`test_honours_gitignored_directories`,
`test_honours_gitignored_file_patterns`, `test_never_descends_into_vcs_metadata`,
`test_finds_scripts_in_nested_directories`, `test_no_gitignore_present_is_tolerated`).

**Changes** (against the real walk-based module): (a) drop the `test-fixtures`
and `test-helpers.sh` clauses from `_keep` (keep only the `workspaces/` clause,
with its justification comment); (b) append the extensionless CLI script after
the walk (the walk's `.sh` filter never matches it), still honouring the gitignore
spec and `_keep`; (c) update the module docstring's scope summary (it currently
says "minus fixtures, jj workspaces, and the sourced-only `test-helpers.sh`") to
"minus `workspaces/` (jj checkouts), plus the explicitly-listed extensionless CLI
script" — leave the walk/`git ls-files` rationale paragraph intact. **Leave the
`shell_sources` walk body byte-for-byte unchanged** (the in-place
`dirnames[:] = [...]` gitignore pruning and per-file filtering); the only edits
are the two `_keep` clauses and the post-walk `out.extend(...)` append. The
elided `...` in the snippet below is *literally the current body* — do not
reconstruct it.

```python
def _keep(rel: str) -> bool:
    """True when a repo-relative shell path should be formatted/linted."""
    if not rel:
        return False
    parts = rel.split("/")
    # workspaces/ holds jj workspace checkouts — full copies of the repo. Linting
    # them would re-lint every tracked script N times over, and they are never
    # edited directly, so they are the one permanent exclusion.
    if parts[0] == "workspaces":
        return False
    return True


# The standalone visualiser CLI is a bash script with no .sh extension (users
# symlink it onto $PATH and type the bare command), so the walk's `.sh` filter
# never matches it. Include it explicitly so it is linted/formatted like any
# other script; shfmt/shellcheck detect bash from its `#!/usr/bin/env bash`.
_EXTRA_SHELL_SOURCES = (
    "skills/visualisation/visualise/cli/accelerator-visualiser",
)


def shell_sources(root: Path | None = None) -> list[str]:
    repo = root or repo_root()
    spec = _ignore_spec(repo)
    out: list[str] = []
    for dirpath, dirnames, filenames in os.walk(repo):
        # ... unchanged walk body: prune gitignored dirs in place, keep .sh
        #     files that are neither gitignored nor excluded by _keep ...
        ...
    # Append the extensionless CLI script(s) — they escape the `.sh` walk filter.
    # Still respect gitignore + _keep (defensive: a future workspaces-relative
    # extra would be filtered, though the current literal never is).
    out.extend(
        s for s in _EXTRA_SHELL_SOURCES
        if _keep(s) and not spec.match_file(s)
    )
    return sorted(out)
```

#### 4. `mise.toml` — `scripts:check` aggregate

**File**: `mise.toml`
**Changes**: Add a per-component aggregate (existing `format:scripts:*` /
`lint:scripts:*` are unchanged). Top-level `format:check`/`lint:check`/`check`
are left alone until Phase 6.

The per-component roll-up is named **`<component>:check`** (`scripts:check`) —
an **entity-first** name, parallel to the existing `version:bump` /
`version:read` / `github:check-auth` tasks (the `<entity>:<action>` form), rather
than the `<family>:<scope>` form (`format:scripts`, `lint:scripts`, `test:unit`)
used by the leaves. This is a deliberate, recognised second namespace shape: the
family leaves are `<family>:<component>:<action>`, and the per-component roll-ups
lead with the component. (An earlier draft justified this by analogy to
`test:unit`, which is wrong — `test:unit` is family-first; the real precedent is
`version:*`/`github:*`.) The same `<component>:check` form is used for
`build-system:check` / `server:check` / `frontend:check` in later phases, and the
distinction is recorded in the Phase 6 folding-rule comment block.

```toml
[tasks."scripts:check"]
description = "Run all shell format and lint checks (shfmt + ShellCheck + bashisms)"
depends = ["format:scripts:check", "lint:scripts:check"]
```

#### 5. `.github/workflows/main.yml` — `check` job → `check-scripts`

**File**: `.github/workflows/main.yml`
**Changes**: Rename the `check` job (lines 81-97) to `check-scripts` and change
its command to `mise run scripts:check`. Update the `prerelease` job's `needs:`
list (line 106) `check` → `check-scripts`.

```yaml
  check-scripts:
    name: Check shell scripts
    runs-on: ubuntu-latest                # lint jobs are ubuntu-only (see note)
    steps:
      - name: Checkout code
        uses: actions/checkout@v5
      - name: Install dependencies
        uses: jdx/mise-action@vX.Y.Z      # exact pin from Phase 1
        with: { install: true, cache: true, experimental: true }
      - name: Run shell checks
        run: mise run scripts:check
```

**Branch-protection required-check name (mandatory, not deferred)**: the GitHub
branch-protection rule that makes `check` a required status check matches by
*job name* and lives in repo settings (not in-repo, so it is invisible to this
plan). Renaming `check` → `check-scripts` will otherwise either block every PR
forever on a check that never reports, or — if matched loosely — silently stop
enforcing (fail-open). So this phase **must** update the required-check name in
branch protection from `check` to `check-scripts`, and verify it by pushing a
deliberately-failing `check-scripts` run and confirming the PR is actually
blocked from merge before the rename is trusted. (This is the concrete form of
the Migration Notes "if staged enablement needs required-check updates" line.)

**Lint-job platform (recorded decision)**: the four new lint jobs stay
**ubuntu-only**, unlike the ubuntu+macOS test matrix. Determinism across a
developer's macOS and CI rests entirely on the exact mise pins resolving to the
identical tool build on both OSes (which is why the exact-pin AC matters); this
is an accepted trade of cross-platform lint coverage for CI cost. Documented
here so the asymmetry is a choice, not an omission.

- Report-only `shellcheck` over the widened set with the new rc; remediate: add

**File**: `tests/unit/tasks/shared/test_sources.py` (extend) or a new

**Changes**: Unlike shell — where `shell_sources()` is a single inspectable
function backed by a regression test — ruff's `extend-exclude` and pyrefly's

   `[tool.pyrefly] project-excludes` from `pyproject.toml`, and assert (a) the

   extension filter and a per-family keep-predicate — out of the shell walk, since

   instead; pass each family's predicate in rather than merging them. A genuine
   shared helper is a reasonable follow-up, not a trivial extraction — duplicating
   a small `.py` walk is acceptable if the abstraction proves awkward.)
2. **Sentinel-in-scope probe**: write a temp file containing a deliberate, unique
   violation **at a real in-scope path** (e.g. `tasks/_coverage_sentinel.py`),

   deliberately introduces the finding.

manual cross-check into a standing, buildable regression guard.

**dist-stub instead of a full frontend build**: `build.rs` only checks that
`frontend/dist/index.html` *exists* (verified — it does not read or embed
contents itself; the embed is a compile-time macro in `src/assets.rs` that is
content-agnostic for a lint compile). Both clippy passes enable `embed-dist`
(pass 1 via `--all-features`, pass 2 via default features), so both need
`dist/index.html` present — but neither needs a *real* SPA. Gating
`lint:server:*` on a full `build:frontend` (node install + Vite production
build) would cross-couple the Rust lint gate to the entire frontend toolchain
and cold-build the SPA on every `check-server` run. Instead, gate on a
lightweight `build:frontend:stub` task that writes a placeholder
`frontend/dist/index.html` **only if one is not already present** (idempotent —
never clobbers a real build):

```toml
[tasks."format:server:check"]
description = "Check Rust formatting with rustfmt (cargo fmt --check)"
run = "invoke format.server.check"

[tasks."format:server:fix"]
description = "Format Rust in place (cargo fmt)"
run = "invoke format.server.fix"

[tasks."build:frontend:stub"]
description = "Write a placeholder frontend/dist/index.html if absent (satisfies embed-dist build.rs for lint-only compiles; not a real build; real build:frontend overwrites it)"
run = "invoke build.frontend-stub"

[tasks."lint:server:check"]
description = "Lint Rust with clippy, two feature passes, -D warnings (gated on a dist stub, not a full SPA build; still two cold compiles — slow on a cold cache)"
depends = ["build:frontend:stub"]            # embed-dist build.rs needs dist/index.html to exist
run = "invoke lint.server.check"

[tasks."lint:server:fix"]
description = "Apply clippy's machine-applicable fixes (cargo clippy --fix, default features only — all-features/dev-frontend arms surface via server:check and may need manual fixes)"
depends = ["build:frontend:stub"]
run = "invoke lint.server.fix"

[tasks."server:check"]
description = "Run all Rust format and lint checks"
depends = ["format:server:check", "lint:server:check"]
```

> **Stub embed mechanics**: under `--all-features` (pass 1) the rust-embed
> `Frontend` struct in `src/assets.rs` is `cfg(not(feature = "dev-frontend"))`,
> so it is compiled *out* and rust-embed never reads `dist/` at all — only
> `build.rs`'s existence check fires. The stub's *contents* are embedded only by
> the default-feature pass 2, where a one-file folder embeds cleanly. So the
> report-only check should confirm pass 2's single-file embed, not assume both
> passes read the folder.
>
> **Implementation contract** (`tasks/build.py` `frontend_stub`, beside
> `build:frontend`): existence-only and atomic —
> `if not (FRONTEND/"dist"/"index.html").exists(): mkdir(parents=True,
> exist_ok=True); write a recognizable placeholder` (e.g. an HTML comment
> `<!-- accelerator lint stub — not a real build -->`). Never overwrite an
> existing file. Unit-test both arms: writes when absent; leaves an existing
> file byte-identical (the no-clobber invariant).
>
> **Shared-artifact safety** (`build:frontend` and `build:frontend:stub` both
> target `frontend/dist/index.html`):
> - **CI is race-free** for two reasons: (a) the `check` aggregate schedules only
>   `server:check` → `lint:server:check` → `build:frontend:stub`; it never
>   schedules a real `build:frontend` (`frontend:check` depends on
>   `deps:install:node`, not the build); and (b) more fundamentally, every CI job
>   runs a *single* aggregate on a *fresh checkout* (`check-server` and
>   `test-unit` are separate jobs on separate runners), so the lint stub and the
>   real `build:frontend` (pulled by `test:unit:visualiser`) never share a tree in
>   CI. The co-scheduling only happens **locally**, when a developer runs an
>   aggregate that combines both — `mise run default` (direct `build:frontend` +
>   transitive stub) or `mise run test`/`default` (where `test:unit:visualiser`'s
>   real build co-schedules with `lint:server:check`'s stub). Blast radius is a
>   local test/dev build only (a transient stub could be embedded into a local
>   embed-dist compile), recoverable by re-running; it never reaches a shipped
>   binary (the release path uses `build:server:cross-compile`, which rebuilds the
>   frontend first and never schedules the stub). The stub's **atomic** write
>   (temp + `os.replace`) prevents a concurrent consumer observing a partial file.
> - **Stale-stub shadowing**: because `build.rs` only checks existence and the
>   stub is only-if-absent, a placeholder left by a prior `lint:server:check`
>   could be embedded by a *later* embed-dist compile in the same tree. ⚠ The
>   exposure is **not** limited to out-of-mise `cargo build`: `build:server:release`
>   (`mise.toml`) is embed-dist but has **no `build:frontend` dependency** and
>   does not write `dist/` — only `build:server:cross-compile` (the CI release
>   path) depends on `build:frontend`. So `mise run build:server:release` in a
>   tree where a prior lint run left a stub would embed the placeholder into a
>   real release binary, silently. (`build:server:release` is a **standalone
>   developer-convenience task** — the automated release pipeline uses
>   `build:server:cross-compile`, which already depends on `build:frontend` — so
>   this is defense-in-depth for the local embed-dist case, not a guard on the
>   shipped artifact.) **Mitigation (required)**: add `build:frontend` to
>   `build:server:release`'s `depends` so a local release always embeds a fresh
>   frontend and overwrites any stub (this also hardens a pre-existing fragility —
>   the task currently relies on the developer having built the frontend first, or
>   it panics in `build.rs`). **Update its `description`** to note it now builds
>   the frontend first (mirroring `lint:server:check`'s dist-stub note) so the
>   added Vite build is documented rather than a surprise. The recognizable
>   placeholder marker remains a
>   secondary detectability aid (VCS revert is *not* a safety net here — the stub
>   lives in gitignored `dist/`, so it never appears in a diff).
> - **`default` race — now a blocking check, not report-only**: add a Phase 6
>   automated success criterion that `mise run default` on a clean tree produces a
>   *real* (non-stub) `dist/index.html` in the final artifact (i.e. `build:frontend`
>   overwrites the stub). If it ever fails, adopt the documented fallback: drop the
>   stub from the `default`/`build:server:release` graphs (CI `check` keeps it), or
>   have `lint:server:*` remove a stub it created. Also treat a **zero-byte**
>   `dist/index.html` as absent in the stub's existence check, so a torn prior
>   write self-heals rather than being embedded.

`tasks/lint/server.py` runs **both** passes (mirrors the two-pass unit tests at
`tasks/test/unit.py`), so the `cfg(not(feature = "dev-frontend"))` arms in
`src/assets.rs` are linted too. It uses `--manifest-path` from the canonical `tasks.shared.paths.CARGO_TOML`
constant rather than a hand-written `context.cd` literal — this *improves on*
`tasks/test/unit.py`, which currently hand-writes the crate path literal (a
worthwhile follow-up is to migrate test/unit.py/integration.py to the same
constant so the manifest-path idiom is uniform). It **runs both passes before
raising** (collect-all-failures, mirroring the `templates` task's explicit
accumulate-then-raise loop) so a developer sees the full diagnostic set in one

```python
from invoke import Context, Exit, task
from tasks.shared.paths import CARGO_TOML

@task
def check(context: Context):
    """Lint Rust with clippy across both feature configs (-D warnings)."""
    passes = {
        # AC-mandated all-features pass (enables dev-frontend):
        "all-features": f"cargo clippy --manifest-path {CARGO_TOML} --all-targets --all-features -- -D warnings",
        # Default-feature pass covers the not(dev-frontend)/embed-dist arms:
        "default": f"cargo clippy --manifest-path {CARGO_TOML} --all-targets -- -D warnings",
    }
    failed = [name for name, cmd in passes.items()
              if context.run(cmd, warn=True, pty=False).exited != 0]
    if failed:
        raise Exit(f"clippy reported findings under feature config(s): {', '.join(failed)}", code=1)
```

tree, so a second pass would error or need `--allow-dirty`. After the single
`--fix`, re-run `check` (both passes) to surface anything that wasn't
machine-applicable. `tasks/format/server.py` → `cargo fmt --manifest-path
{CARGO_TOML} --all -- --check` / `cargo fmt --manifest-path {CARGO_TOML} --all`.

`tasks/format/__init__.py` / `tasks/lint/__init__.py` `from . import server`

```toml
depends = ["types:frontend:check", "types:build-system:check"]

[tasks.fix]
description = "Apply all formatters + safe lint fixes (MECHANICAL only — no type-checks, and shell has no autofixer; you must still run `mise run check` before pushing)"
depends = ["format:fix", "lint:fix"]

[tasks.check]
description = "Run all read-only format, lint, and type checks (what CI runs)"
depends = ["frontend:check", "server:check", "build-system:check", "scripts:check"]
```

(Each per-component phase already created its `<component>:check` **and wired its
CI job into `prerelease.needs`**; this phase only rewrites the four shared
family aggregates left untouched earlier, adds the top-level `fix` convenience
aggregate — the single "clean up my diff before pushing" command the tree
otherwise lacked — and repoints `default`.)

**`default` task** (`mise.toml:220-222`) is currently `depends =
["build:frontend", "build:server:dev", "format:fix", "lint:check", "test"]`.
After this rewrite `lint:check` no longer transitively covers type-checks (they
moved to the `types:*` family), so the bare `mise run` would silently skip them
— a "passes locally, fails in CI" gap. **Add `types:check`** so the local
default loop catches everything CI does:

```toml
[tasks.default]
description = "Full local CI mirror: build, format-fix, lint, type-check, test (heavy — reformats in place + compiles Rust several times; for a fast inner loop use a single <component>:check)"
depends = ["build:frontend", "build:server:dev", "format:fix", "lint:check", "types:check", "test"]
```

  learn-once: "`<component>:check` runs every applicable family for that

  linters. Consult `mise tasks` for the per-component leaves." Also note two shape

  entity-first like `version:*`) but **trails** in the families

  are `<component>:check` roll-ups but **no `<component>:fix`** — fix one component
  via `format:<component>:fix` + `lint:<component>:fix`, or the top-level `fix`.
  This converts the depth/shape asymmetries from surprises into documented behaviour.
- **Document the pre-push loop** (CI-only enforcement means no hooks catch a
  violation before push): the `mise.toml` comment block is the detailed
  reference, but it is **not surfaced by `mise tasks`** and the current

  its `format:<component>:fix` and `lint:<component>:fix` (there is no

  "python" must still land on the component. The `build-system:*` task
  descriptions all name Python/ruff/pyrefly (so `mise tasks | grep -i python`

## Testing Strategy

### Unit / regression tests:
- `shell_sources()` scope assertion — **invert the existing assertions in
  `tests/unit/tasks/shared/test_sources.py`** (Phase 2), don't add a new file.
- `TestShellcheckTask.test_command` updated to the bare-`shellcheck` prefix
  (Phase 2).
- Python coverage guard — ruff/pyrefly file set equals the VCS-agnostic `.py`
  walk minus justified excludes (Phase 3 §5; not `git ls-files`, which is
  jj-workspace-blind).
- `_render_cargo_toml` preserves `[lints.clippy]` **and a verbatim rationale
  comment** (Phase 4) — guards the version templating.
- Existing suites must stay green after each sweep: `test:unit:tasks`,
  `test:integration:tasks` (Python), `test:unit:visualiser` /
  `test:integration:visualiser` (Rust fmt), `test:unit:frontend` **plus
  `test:e2e:visualiser` + visual-regression** (Biome reformat — the largest
  sweep), `test:integration:migrate` / `test:integration:github` (shell
  reformat).

### Falsification probes (per component, the AC's CI verification):
For each component, introduce **one representative violation per check command**,
push, observe the corresponding `check-<component>` job exit non-zero, link the
run from the PR, then revert:
- scripts: shfmt drift + shellcheck finding + bashism (place ≥1 probe in a
  newly-in-scope former-fixture file).
- build-system: ruff lint + `ruff format` drift + pyrefly type error.
- server: `cargo fmt` drift + clippy warning.
- frontend: Biome lint + format drift + tsc type error.

Note: a probe proves the check command is **wired and blocking**, not that the
file-set is complete — an in-tree violation would flag even if the tool were
checking too few files. File-set *breadth* is verified separately by the
`shell_sources()` scope test and the Python coverage guard.

### Report-only checkpoints (per component, the sizing contingency):
Before each sweep, run the tool(s) in report mode and record counts in the PR. If
any component's remediation is unmanageable as a handful of commits, that
component's slice already is the per-language child the contingency calls for —
no re-scope needed (this plan is already per-component).

## Performance Considerations

- The `check-server` job cold-compiles Rust twice (two clippy feature passes) with
  no cargo cache — minutes of CI time. Acceptable; caching is explicitly out of
  scope. The lightweight `build:frontend:stub` runs first (mise `depends`) — a
  near-instant placeholder-write, **not** the full `build:frontend` Vite build,
  which decouples the Rust lint gate from the frontend toolchain.
- Biome's `project` domain walks the whole frontend module graph — heavier than
  lint-only, but the frontend is small.
- pyrefly/ruff/shellcheck/shfmt are fast on this corpus (63 `.py`, 173 shell).

## Migration Notes

- Phases land as separate per-component PRs (work item default). Each is
  independently mergeable and — because each PR adds its job to `prerelease.needs`
  and updates the branch-protection required-check name — makes its component's
  check blocking the moment it lands, with no grandfathered violations at any
  blocking moment.
- Sweep ordering & broadcast: before each sweep PR lands, enumerate in-flight
  work in that language area (or agree the window with owners) **and broadcast
  the imminent reformat to the wider team** — the frontend sweep alone rewrites
  ~268 files and will conflict with essentially every open frontend branch.
  In-flight branches should **rebase and then re-run `mise run format:<component>:fix`
  / `lint:<component>:fix`** to regenerate the mechanical state deterministically,
  rather than hand-resolving reformat conflicts (which risks re-introducing old
  style the now-blocking check rejects).
- Branch-protection **is** changed by this item: the `check` → `check-scripts`
  rename (Phase 2) requires updating the required-check name, and each new
  per-component job (Phases 3–5) must be added as a required check as it becomes
  blocking. Main CI's required status is otherwise pre-existing. (This supersedes
  the earlier "no branch-protection change expected" assumption — the rename
  makes the required-check-name update mandatory; see Phase 2.)
