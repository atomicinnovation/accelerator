## Plan Review: CHANGELOG 1.21.0 Consolidation and Cleanup

**Verdict:** REVISE

The plan is an unusually well-grounded documentation cleanup: it specifies the
rewritten `[1.21.0]` section verbatim, traces every retained claim to a verified
research document, foregrounds the previously-missing migration story (0005/0006),
and correctly collapses the three-way Playwright MCP contradiction into a single
`Removed` narrative. The curated content itself is largely accurate and
audience-appropriate. The reason for REVISE is not the prose — it is the
**Phase 2 verification harness**, which is load-bearing for the whole plan and
is currently near-worthless: the audit script has no `set -e`, so ~11 of its ~13
assertions cannot fail the run and it prints `AUDIT OK` unconditionally; and one
of those assertions directly contradicts the content the plan tells the
implementer to write. Secondary themes: the line-number-anchored edit is fragile
at the `[1.20.0]` seam with no automated confinement guard, and the aggressive
audience-filtering dropped several genuine user-action breaking-change notes.

### Cross-Cutting Themes

- **The Phase 2 audit gives false confidence** (flagged by: Correctness,
  Test Coverage) — No `set -e`/shebang means every bare `! grep`/`grep -q`
  assertion executes but cannot gate; the script falls through to
  `echo "AUDIT OK"` regardless. On top of that the audit has self-contradicting
  assertions, exact-wording-pinned greps that miss reworded variants, and no
  coverage for several research-enumerated removals. This is the single most
  important thing to fix — without it, the plan's primary success criterion
  ("Content audit prints AUDIT OK") proves almost nothing.
- **Edit confinement at the `[1.20.0]` seam is fragile and under-verified**
  (flagged by: Correctness, Safety, Test Coverage) — The edit is anchored by
  absolute line numbers ("replace lines 5-411") whose two stated boundaries
  diverge by one (line 412 is a blank separator; `## [1.20.0]` is at 413), and
  the "byte-for-byte unchanged" guarantee for everything outside `[1.21.0]` is
  checked only by a human reading `jj diff`. The automated parse check passes
  even if the adjacent frozen section is clipped.
- **Dropping the dedicated `### Breaking` section costs breaking-change
  discoverability** (flagged by: Usability, Documentation) — Three real
  user-action items (Node ≥ 20 + MCP re-registration, the `[[TICKET-NNNN]]`
  wiki-link break, and the session-restart-then-migrate ordering) are buried in
  prose, scattered across three locations, or dropped entirely. An upgrader who
  scans headings and bold lead-ins will miss them.
- **The plan overstates what its tooling guarantees** (flagged by: Standards,
  Test Coverage) — `scripts/test-format.sh` is repeatedly framed as a general
  format/hyphenation gate; it actually checks only three narrow `work item`
  patterns and self-describes as "intentionally targeted, not exhaustive."

### Tradeoff Analysis

- **Audience-filtering (Documentation) vs. user-action completeness
  (Usability)**: The plan's removal of contributor-only detail is a real
  strength, but the cut went one notch too far in places. Node ≥ 20, MCP
  re-registration, the wiki-link break, and the session restart are *user
  actions*, not contributor detail — they should survive the cut even if stated
  tersely. Recommendation: keep user-action items; drop only mechanism detail.
- **Strict Keep-a-Changelog (Standards) vs. breaking-change scannability
  (Usability)**: Dropping `### Breaking` is standards-compliant, but costs
  upgraders a scan target. Keep-a-Changelog permits inline `**Breaking:**`
  bold lead-ins within `Changed`/`Removed`; that reconciles both — or hoist the
  imperative into the top callout.
- **`mcp__playwright__*` jargon in the Removed narrative**: Test Coverage flags
  it as a contributor token to strip; but a "we removed MCP" entry arguably
  benefits from naming what it replaced. This is a judgment call for you — purity
  vs. explanatory clarity.

### Findings

#### Critical

- 🔴 **Correctness + Test Coverage**: Audit script has no `set -e`; ~11 of ~13 assertions cannot fail the run
  **Location**: Phase 2, Section 1 — Bespoke content audit (script lines ~354-395)
  The script declares no `set -e` (and no shebang). Only the two `for` loops use `|| { echo FAIL; exit 1; }`; the eight bare `! grep -qE …` absence checks and three bare `grep -q …` presence checks have no exit-on-failure, so a failed assertion's non-zero status is ignored and execution falls straight through to the unconditional `echo "AUDIT OK"`. The "Content audit prints AUDIT OK" success criterion is therefore near-tautological.

- 🔴 **Correctness + Documentation**: An absence assertion contradicts the curated content the plan specifies
  **Location**: Phase 2 audit (plan line ~377) vs. Phase 1 Changed/0004 (plan lines ~244-247)
  The audit asserts `! grep -qE 'paths\.design_(inventories|gaps)\b'`, but the curated Changed entry deliberately writes "The config keys `paths.research`, `paths.design_inventories`, and `paths.design_gaps` are renamed to …". The `\b` boundary matches (a backtick follows), so the true result is FAIL on the plan's own output. Naming the old keys is the correct way to document a rename; the assertion is mis-specified. Once `set -e` is added, this would block a correct implementation.

#### Major

- 🟡 **Documentation**: Removed entry conflates the ~150 MB Chromium install size with the ~500 MB free-disk floor
  **Location**: Phase 1 curated content — Removed (Playwright MCP)
  The entry says runtime/hybrid needs "~500 MB free disk for the first-run Chromium install." The install is ~150 MB (`ensure-playwright.sh`, and the research); 500 MB is the free-disk floor the bootstrap checks (`DISK_FLOOR_MB=500`). Merging them overstates the install footprint ~3.3× to an upgrader.

Verified against the tree: line 411 is the last `[1.21.0]` content, line 412 is a blank separator, `## [1.20.0]` is at line 413. So "replace 5-411" keeps the blank separator, but the semantic reading ("down to the `## [1.20.0]` line" = 5-412) consumes it, leaving the `> Note` blockquote directly abutting `## [1.20.0]` with no blank line — risking keepachangelog mis-attributing the heading.

The curated content already contains "0003–0006" in the top callout and the Migrations intro, so each check passes even if a per-migration `### Migrations` bullet (the exact gap the research flagged for 0005/0006) were missing or truncated.

  **Location**: Phase 2 audit (absence checks) vs. Phase 1 Removed (curated content)
  The research lists `mcp__playwright__*` as contributor-internal tokens to eliminate, but there is no grep for them. The curated Removed bullet itself still says "instead of `mcp__playwright__*` tools, eliminating the sub-agent MCP-inheritance failure mode" and "the project-scoped `@playwright/mcp` server." Either decide this jargon is permitted in the Removed narrative (defensible) or strip it — but the audit currently neither asserts nor reflects a decision.
