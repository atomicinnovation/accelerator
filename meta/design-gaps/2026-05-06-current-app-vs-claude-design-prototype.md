# Design Gap Analysis: current-app → claude-design-prototype

## Overview

This analysis compares the running Accelerator Visualiser frontend (React 19 SPA, captured at git commit `94a0ed6` on 2026-05-06) against the Claude design prototype (a single-file JSX-driven mock served from `claudeusercontent.com`, captured the same day). The current app is a fully functional implementation with twelve doc-type list views, per-document detail screens with markdown rendering, lifecycle clusters, and a drag-and-drop kanban board. The prototype proposes a substantially more developed visual language: a layered token system (`--atomic-*` brand palette, `--ac-*` semantic surface layer, light/dark themes), a top-bar app shell with breadcrumbs and server-origin pill, lifecycle-phase-grouped sidebar navigation, an activity feed, a multi-form pipeline visualisation, and a richer set of decorated components. The prototype is not feature-complete — per-document detail screens, drag interactions, search results, sort/filter overlays, and the toast lifecycle were not exercised — so this gap analysis flags those omissions as TBD rather than removals. We need to treat every drift category below as a new prototype-defined surface that the current app must adopt, with the per-doc-detail screens being the one major exception where the current app already has working capability the prototype lacks and which must be retained.

## Token Drift

The current app exposes only eight named CSS custom properties for colour and relies on roughly fourteen hard-coded hex values scattered across CSS modules; the prototype defines a layered token system with a brand palette (`--atomic-*`), legacy semantic aliases (`--fg-*`, `--bg-*`, `--accent`, `--stroke`), and an active semantic surface layer (`--ac-*`) with full light/dark overrides under `[data-theme]`. We need to migrate every hard-coded hex listed in `current-app` Design System / Tokens (e.g. `#111827`, `#1f2937`, `#374151`, `#6b7280`, `#d1d5db`, `#f3f4f6`, `#f9fafb`, `#2563eb`, `#1d4ed8`, `#dbeafe`, `#6366f1`, `#fef3c7`, `#f59e0b`, `#fef2f2`, `#fecaca`, `#991b1b`) onto the prototype's named `--ac-*` tokens (e.g. `--ac-fg-strong`, `--ac-fg-muted`, `--ac-stroke`, `--ac-bg-card`, `--ac-bg-sidebar`, `--ac-bg-active`, `--ac-accent`, `--ac-warn`, `--ac-err`) so the codebase stops carrying its de-facto palette inline.

The current app has no formal typography scale: body and chrome use `font-family: system-ui, sans-serif`, code uses `monospace`, and font sizes are set ad hoc per component. The prototype defines a four-family stack (`Sora` display, `Inter` body, `Fira Code` mono, `Raleway` support) plus an eleven-step size scale (`--size-xxs: 12px` through `--size-hero: 68px`), four line-height tokens (`--lh-tight 1.05`, `--lh-snug 1.2`, `--lh-normal 1.5`, `--lh-loose 1.6`), and a `--tracking-caps: 0.12em` token for uppercase eyebrow labels. We need to introduce these typography tokens into `src/styles/global.css` and `src/styles/tokens.ts`, load the four web fonts, and replace every inline `font-family` / `font-size` declaration in component CSS modules with token references so headings, body, and eyebrow labels share a single scale.

The current app has no formal spacing or radius scales — each module hard-codes `px` and `rem` values inline, and recurring border-radius values (`2px`, `3px`, `4px`, `6px`, `8px`, `9999px`) are repeated without naming. The prototype defines an eleven-step spacing scale (`--sp-1: 4px` through `--sp-11: 124px`) and a four-step radius scale (`--radius-sm: 4px`, `--radius-md: 8px`, `--radius-lg: 12px`, `--radius-pill: 999px`). The system must adopt these scales as the canonical source for layout spacing and corner geometry, and every CSS module must consume them via `var(--sp-N)` and `var(--radius-*)` rather than literal values.

The current app has only one shadow rule (`box-shadow: 0 1px 4px rgba(29, 78, 216, 0.12)` on lifecycle card hover) while the prototype defines five elevation tokens covering both static and theme-swapping shadows: `--shadow-card`, `--shadow-card-lg`, `--shadow-crisp`, plus per-theme `--ac-shadow-soft` and `--ac-shadow-lift`. We need to add a shadow token tier so cards, raised surfaces, and lifted overlays each have a named elevation that swaps correctly between light and dark themes.

The current app has no theming mechanism at all — there is one and only one colour palette, no `data-theme` attribute, and no dark-mode tokens. The prototype provides a complete dark-theme override of every `--ac-*` colour and shadow token under `[data-theme="dark"]`, plus a `[data-font="mono"]` font-mode swap that repoints display and body to Fira Code. The system must support a `data-theme` attribute on the document root and expose a theme toggle in the topbar so users can switch between light and dark without reloading.

## Component Drift

The current app's `Sidebar` partitions doc types into three flat groups (Documents / Views / Meta) and renders the navigation as plain text links; the prototype's `Nav` component partitions doc types by lifecycle phase (Define / Build / Ship / Remember) under a single LIBRARY heading, decorates each item with a per-doc-type `Glyph`, a count badge, and an unseen-changes dot. Users need a redesigned Sidebar nav whose grouping reflects the research → plan → implement workflow rather than the storage-shape distinction between virtual and concrete doc types, with each nav item showing a per-type icon, an item count, and an unseen-changes indicator.

The current app has no Topbar — the `RootLayout` is a two-column shell of sidebar plus main scroll area, and SSE / version status is rendered inside the sidebar footer. The prototype introduces a persistent `Topbar` component with brand wordmark, centred breadcrumb crumbs, server-origin pill (`127.0.0.1:NNNNN` plus a green pulse), `SSE` connectivity indicator, and a theme toggle. Implement Topbar as a new top-level chrome component owned by `RootLayout`, exposing breadcrumbs that derive from the active TanStack route, the server origin from `useServerInfo`, the SSE state from `useDocEventsContext`, and the theme toggle from a new theme-context hook.

The current app's chip vocabulary is limited to `FrontmatterChips` (definition-list pairs surfaced from a document's frontmatter) and a malformed-frontmatter banner. The prototype defines a generic `Chip` component with five named variants (`--green` for done/accepted, `--indigo` for in progress / live, `--amber` for approve-with-changes, `--neutral`, `--sm`) used across kanban cards, lifecycle cards, library tables, and templates indicators. We need a generic Chip component with these five variants so status pills can be reused across screens rather than being open-coded per surface.

The current app's `LibraryTypeView` renders a sortable table where sort is driven by clicking column headers, and there is no filter UI. The prototype renders the same table with two extra page-level controls — an `.ac-sort-btn` pill ("Recently modified") and a `.ac-filter` pill — plus a structured `Page` wrapper with eyebrow row, `<h1>`, subtitle/count, and right-aligned actions. Users need a sort-button and filter-button pair surfaced in the page header alongside an eyebrow / title / subtitle structure, replacing today's bare `<h1>` and column-click-only sort.

The current app's `PipelineDots` is a single component rendering an eight-dot completeness row inside lifecycle cluster cards; the prototype renders the eight-stage pipeline three different ways: a full `.ac-hexchain` (linked stage tiles with labels) on lifecycle cluster detail, a compact strip (`.ac-lcard__pipe`) on lifecycle cluster cards, and a micro `.ac-stagedots` row embedded in kanban cards. We need to extend the pipeline visualisation into three rendering modes — full hexchain, compact strip, and micro dots — and surface the appropriate mode on lifecycle cluster detail, lifecycle index, and kanban respectively.

The current app's `WorkItemCard` (the draggable kanban card) shows the work-item ID, mtime, title, and an optional `frontmatter.type` field. The prototype's `.ac-kcard` adds an embedded `.ac-stagedots` micro-pipeline showing each card's lifecycle completeness, an `.ac-kcard__links` "N linked" relation count, and richer ID prefixing (e.g. `PROJ-NNNN`, `ENG-NNNN`, `META-NNNN`). We need to extend the WorkItemCard to embed pipeline-completeness micro-dots, surface the linked-artifact count, and render namespaced work-item IDs alongside the existing chrome.

The current app has no doc-type icon system — items are distinguished by text labels alone. The prototype defines a `Glyph` component that renders a square doc-type icon at multiple sizes (16/24/32px) with per-doc-type fill colours (red Decision, orange Research, blue Plan, purple Plan review, green Validation, teal PR, mauve PR review, red-pin Note, dark-red Work item) and embeds it inside nav items, page eyebrows, kanban cards, timeline steps, lifecycle cards, and activity items. Implement Glyph so every doc-type reference across the app renders with a consistent icon and colour, then thread it through Sidebar nav, Page header eyebrows, kanban cards, lifecycle cards, timeline steps, and activity-feed rows.

The current app has no toast / notification component. The prototype includes a `Toaster` (`.ac-toaster`) ephemeral notification dialog with icon, heading, message, and close button, demonstrated by the "External edit detected · A reviewer agent updated `WORK-0007`" toast on initial load. Implement Toaster as a new ephemeral notification component, mounted at the root layout, that surfaces external-edit events received via SSE so users see when an out-of-band agent has modified a document they are viewing.

The current app has no activity-feed component. The prototype defines an `Activity` feed (`.ac-activity`) inside the sidebar that shows the five most recent file-change events as rows, each with a per-doc-type `Glyph`, a "type · action" line (created / edited / moved-to-status), a relative timestamp + filename line, and a `LIVE` badge in the heading. Implement Activity as a new sidebar block subscribed to the existing SSE `/api/events` stream, rendering rolling file-change events with the per-doc-type glyph, action verb, relative timestamp, and a live badge.

## Screen Drift

The current app's lifecycle cluster detail (`/lifecycle/{slug}`) renders only a vertical timeline of pipeline stages; the prototype renders the same timeline preceded by a full `.ac-hexchain` strip at the top of the page that visualises the same eight-stage pipeline in linked-tile form with stage labels. The system must augment the lifecycle-cluster screen with a hexchain header strip above the existing timeline, sharing pipeline data with the per-stage cards below.

The current app's `/library` route redirects straight to `/library/decisions`, so there is no library-overview hub; the prototype provides a `library-overview` screen that groups the twelve doc types by lifecycle phase (Define / Build / Ship / Remember), with each group containing per-doc-type cards that show an icon, the doc-type label, the doc count, and a "latest · {title}" preview. We need to add a Library overview hub at `/library` that replaces the current redirect, presenting all twelve doc types as drillable phase-grouped cards with counts and latest-document previews.

The current app's lifecycle-index sort affordance is a three-mode toggle (Recent / Oldest / Completeness) plus a free-text filter input; the prototype's lifecycle-index uses a two-segment pill control (`Updated` / `Completeness`) and no filter input. We need to confirm with stakeholders whether the prototype's reduced sort set and the removal of the filter input are intentional simplifications or omissions, before changing the lifecycle-index controls.

The current app's kanban columns are derived dynamically from the server-side `kanban-config` endpoint (configurable per-workspace), while the prototype's kanban-view shows a fixed three-column board (Todo / In progress / Done) with a `live` chip and `N total` count in the page subtitle. We need to confirm that the prototype's three-column board is illustrative rather than prescriptive — preserving the configurable column set is required by ADR-0024, so the redesign must keep dynamic columns while adopting the prototype's `live` / `N total` page-subtitle treatment.

The current app's templates view (`/library/templates/{name}`) renders three stacked tier panels with an active-tier marker plus the rendered template body. The prototype's `templates-view` adds an inline tier-presence row on the index list (each tier shown with a `present`-state colour and a "winning" highlight), expands a chosen template into a stacked tier card alongside a separate right pane that renders the template body with a `sha256-…` etag header. The system must redesign the templates index to surface per-tier presence inline on each row and add a sha256 etag header to the rendered tier preview pane on the detail screen.

## Net-New Features

The prototype introduces light / dark theming via a `data-theme` attribute on the document root, with a topbar `Toggle theme` button cycling between values and a complete `--ac-*` override block under `[data-theme="dark"]`. Implement Theming as a context-backed hook that sets `data-theme` on `<html>`, persists the user choice (e.g. to `localStorage`), and exposes a Toggle button in the new Topbar; wire it through the token migration so every component uses `--ac-*` tokens that swap automatically.

The prototype introduces a font-mode swap via `[data-font="mono"]` that repoints `--ac-font-display` and `--ac-font-body` to Fira Code, exposing a "monospace mode". Implement FontMode as an attribute-toggling hook (mirroring the theme toggle) so users can switch between display and mono presentations of the entire app once typography tokens are in place.

The prototype introduces a persistent sidebar search box ("Search meta/…") with a `/` keyboard-shortcut hint chip. Implement Search to expose a sidebar search input with a `/` keybind activator, hooked to a forthcoming `/api/search` endpoint that returns matches across all twelve doc types and surfaces the results inline.

The prototype introduces an unseen-changes dot — a small accent dot rendered next to a nav item label when that doc type has had file changes since the user's last visit (observed on Plans `· 18` and Decisions `· 9` in the prototype). Implement UnseenChangesIndicator as a per-doc-type tracker that listens to SSE events, marks doc types as "unseen" until their list view is opened, and feeds the `Nav` component so each item can display the dot.

The prototype introduces an external-edit toast that fires when an SSE event indicates another agent (e.g. a reviewer) has modified a document the user is currently viewing — formatted as "External edit detected · A reviewer agent updated `WORK-0007` while you were looking at it. Query invalidated." Implement ExternalEditToast that subscribes to the existing SSE event stream, correlates incoming change events against the active document route, and surfaces a Toaster notification with the affected ID and a "Query invalidated" message when a match is found.

The prototype introduces a server-origin pill in the topbar (`127.0.0.1:NNNNN`) with a green pulse, accompanying a separate `SSE` indicator. Users need a topbar pill that surfaces the live server origin from `useServerInfo` alongside a connectivity-pulse animation, replacing today's text-only sidebar-footer status.

## Removed Features

The current app provides per-document detail screens at `/library/{type}/{slug}` rendering frontmatter chips, the markdown body via `MarkdownRenderer`, a related-artifacts aside (Targets / Referenced by / Same lifecycle), and inline "Document not found." error states. The prototype contains no per-document detail screen at all — clicking a row in `library-plans` / `library-decisions` is a no-op. We need to confirm with stakeholders that the absence of detail screens in the prototype is a known TBD (the prototype crawler explicitly flagged it as such) rather than an intent to remove the detail-screen capability — the existing detail surface should be retained and have the new design system applied.

The current app's `MarkdownRenderer` integrates `react-markdown` + `remarkGfm` + a custom `remarkWikiLinks` plugin that rewrites `[[ADR-N]]`, `[[WORK-ITEM-N]]`, and `[[WORK-ITEM-PROJ-N]]` markers into anchor links, pending markers, or unresolved markers. The prototype demonstrates markdown rendering only inside the templates tier-preview pane and shows no wiki-link affordance. We need to confirm with stakeholders that the wiki-link rewrite remains a required capability before any redesign, since cross-document navigation via `[[…]]` is load-bearing for the existing detail screens.

The current app surfaces a malformed-frontmatter banner inside per-document detail when the backend reports a YAML parse failure (`state === 'malformed'`). The prototype includes no analogous banner because it has no detail screen. We need to confirm that the malformed-frontmatter banner is to be retained in the redesigned detail screen, since it is an authored signal that depends on backend state.

The current app's `useDeferredFetchingHint` exposes a 250ms-debounced "Updating…" hint to suppress flicker on fast re-fetches inside the related-artifacts aside on detail pages. The prototype shows no analogous hint because it lacks the surfaces that consume it. We need to confirm whether the deferred-fetching hint is to be preserved when the detail screen is redesigned.

The current app's lifecycle-index has a free-text filter input above the cluster grid; the prototype removes it in favour of a Updated / Completeness segmented sort only. We need to confirm with stakeholders whether removing the filter input is intentional, since the current filter is the only way to narrow large lifecycle lists by slug substring.

## Suggested Sequencing

The token migration is the load-bearing prerequisite — until `--ac-*` colour, typography, spacing, radius, and shadow tokens exist in `src/styles/`, no component re-skin can land without immediately diverging from the target. We need to sequence the work so that token introduction happens first as a self-contained pass that adds the new tokens alongside the existing `--color-*` set, followed by a per-component sweep that replaces hard-coded values with token references. Theming and font-mode swaps should land immediately after the token pass, since they are CSS-only once the tokens cascade through `[data-theme]`. Component changes that depend on the token system (Topbar, redesigned Sidebar nav, generic Chip, Glyph, Page wrapper, expanded WorkItemCard, three-form pipeline visualisation) can then proceed in parallel as each component is refit. Net-new features (Search, ActivityFeed, UnseenChangesIndicator, ExternalEditToast, Toaster) should follow the chrome work, since they each need either the new Topbar or the new Sidebar to surface. Per-document detail screen redesign and the screen-level drift items (library-overview hub, lifecycle hexchain header, templates inline tier presence) should come last, since they depend on Glyph, Chip, and the Page wrapper being available. Throughout, the items in Removed Features that ask for stakeholder confirmation must be resolved before the corresponding screen-redesign work begins, so scope is fixed before implementation.
